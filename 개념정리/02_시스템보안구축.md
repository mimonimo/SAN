# 시스템 보안 구축 — 개념 정리

> 학습모듈 목표: 시스템 보안요구사항을 정의하고 안전한 정보시스템을 설계·구현·검증할 수 있다.

---

## 학습 1. 시스템 보안 설계하기

### 1-1. 시스템 보안요구사항 정의

#### 시스템 보안 개념
- 시스템 보안의 정의 : 시스템 구성 요소·자원의 기밀성·무결성·가용성을 보장하기 위해 취해지는 활동. 외부 피해(해커·바이러스·자연재해)와 내부 피해(불만·부정직·해고당한 종업원)로부터 조직의 컴퓨터 시스템·데이터베이스·네트워크 등 정보 자원을 보호하는 방법.

#### 시스템 보안 범위
- 개념 : 시스템 보안은 주로 계정·패스워드 관리, 세션 관리, 접근제어, 권한 관리, 로그 관리 및 취약점 관리, DBMS 관리 등을 포함한다.

**1. 계정** : 적절한 권한을 가진 사용자를 식별하기 위한 가장 기본적인 인증 수단. 계정(account)이란 한 사용자에게 소유되는 모든 파일·자원·정보를 말함. 계정과 패스워드 관리가 보안의 시작.

**2. 패스워드**
- 패스워드의 정의 : 허가 없이 DB·파일에 접속하는 것을 방지하기 위한 안전 대책으로, 사용자 ID와 함께 입력하여 정당한 사용자임을 식별하도록 주컴퓨터에 전달하는 고유의 문자열.
- 패스워드의 기능 : 온라인 정보 서비스는 패스워드가 없으면 이용 불가. 주컴퓨터가 입력된 ID·패스워드를 등록된 것과 대조하여 일치 시 접속 허용.

**3. 세션 관리**
- 세션의 개념 : 망 환경에서 사용자/컴퓨터 간 대화를 위한 논리적 연결, 또는 프로세스 간 통신 시 메시지 교환으로 서로를 인식한 이후부터 통신을 마칠 때까지의 기간.
- 세션 관리의 개념 : 활성화된 접속에 대한 관리로, 일정 시간 경과 시 세션을 종료하고 비인가자의 세션 가로채기를 통제.

**4. 접근제어**
- 접근의 개념 : 데이터가 있는 곳을 알아내어 가져오는 것, 또는 사용자가 시스템·프로그램을 이용하기 위해 최초로 접속을 시도하는 것.
- 접근제어의 개념 : 시스템이 다른 시스템으로부터 적절히 보호되고 꼭 필요한 사람에 의해서만 필요한 서비스를 제공하도록 전체 시스템 차원에서 접근을 통제하는 것.

**5. 권한 관리**
- 권한의 개념 : 직책 수행에 필요한 결정권 등을 합한 것. 조직 내 위치로 인해 갖는 공식적인 힘이며 합법성이 강조됨. 한 개인·집단을 지배할 수 있는 권리(right).
- 권한 관리의 개념 : 각 사용자가 적절한 권한으로 적절한 정보 자산에 접근하도록 통제하는 것.

**6. 로그 관리**
- 로그의 개념 : 데이터를 기록한 것 또는 데이터의 인쇄된 기록.
- 로그 관리의 목적 : 시스템 내부·외부에서 시스템에 영향을 미칠 경우 해당 사항을 기록하여 문제 해결·예방 활동에 활용.

**7. 취약점 관리**
- 취약점 개념 : 공격자가 시스템의 정보 보안을 낮추는 데 사용되는 약점. 시스템 민감성·결함, 결함에 대한 접근, 결함에 대한 공격 가능성을 의미.
- 취약점 관리의 개념 : 취약점을 확인·분류·치료·완화하는 주기적인 과정.
- 취약점 관리의 목적 : 계정·접근제어 등이 잘 갖춰져도 시스템 자체의 결함으로 보안 문제가 발생할 수 있으며, 이런 결함을 체계적으로 관리하는 것.

**8. DBMS 관리**
- DBMS 개념 : 기업·조직이 보유하는 데이터베이스를 저장·관리하는 시스템(Data Base Management System). 전사적 자원·서비스 데이터를 포함하므로 보안이 중요.
- DBMS 관리의 목적 : DBMS 계정 관리, DB 접근 권한, DB 암호화, DB 백업 등을 관리.

### 1-2. 시스템 보안 설계

#### 시스템 보안 설계의 개념
- 시스템 보안 설계 개념 : 악성코드·개인정보 유출·중요 정보 노출·비밀 데이터 손실 등의 비즈니스 문제를 방지·해결하기 위해 안전한 정보시스템 환경을 구축하고자 관리적·기술적·물리적 보안 차원에서 보안 구축 계획을 수립하고 보안 설계서를 작성하는 것.

**시스템 보안 설계 대상** : 계정 관리, 패스워드 관리, 세션 관리, 접근 통제, 권한 관리, 취약점 관리(각 개념은 1-1과 동일).

**시스템 보안 설계 원칙**
- 원칙 1 : 자원에 적용되는 보안 수준은 조직에 주는 가치에 비추어 적절해야 하며, 보안 비용은 그 혜택에 비추어 합리적이어야 한다.
- 원칙 2 : 보안 아키텍처는 변화하는 보안 필요·요구사항을 수용하고, 보호 레벨 변화 시에도 기본 아키텍처 수정 없이 지원하며, 미래 확장을 수용할 만큼 확장성이 있어야 한다.
- 원칙 3 : 조직이 안전한 업무를 전자적으로 수행하도록 통합된 보안 서비스를 제공해야 한다.
- 원칙 4 : 모든 컴퓨터 플랫폼에 걸쳐 일관성 있는 프레임워크를 제공해야 한다.

#### 정보시스템 보안 계층
- Client : 사용자 단말(단말기, PC 등)에 대한 보안 설정 및 적용 기술.
- Network : 네트워크 시스템(라우터, 브릿지, 스위치, 보안 장치 등) 및 네트워크 보안 솔루션에 대한 보안 설정 및 적용 기술.
- Server : 서버 시스템의 보안 설정 및 적용 기술.
- Application : 업무 관련 애플리케이션에 대한 보안 설정 및 적용 기술.
- Data(DB) : 저장 정보(주요 정보)에 대한 보안 설정 및 적용 기술.

#### 정보보호 기술 영역
- 인증(기술 영역) : 정보 자산에 접근하고자 하는 사용자와 장비에 대한 접근 중요도에 따라 차별화된 인증 방안을 적용하는 영역.
- 접근 통제(기술 영역) : 사용자 및 장비의 접근 필요성에 따라 정보 자산에 대한 접근권한을 부여함으로써 비인가자의 무단 접근을 제한하는 영역.
- 암호화(기술 영역) : 개인정보·인증정보 등 기밀성을 요구하는 정보를 변조·삭제·오남용 등으로부터 보호하기 위하여 전송 및 저장 시 암호화를 적용하는 영역.
- 로깅 감사(기술 영역) : 사용자 및 장비의 정보 자산에 대한 접근·처리 내역을 보존하여 감사 추적성 및 모니터링 증적을 확보하는 영역.
- 프라이버시(기술 영역) : 개인정보의 유출 및 오남용 방지를 위해 주요 경로에 대한 보호 대책을 적용하는 영역.
- 침해사고 예방 및 대응(기술 영역) : 네트워크·시스템·데이터의 보안을 위협하는 침입 행위를 탐지·차단·예방하기 위한 방안을 적용하는 영역.

### 1-3. 시스템 보안 구현 준비

#### 시스템 보안 구현을 위한 환경 개요
- 시스템 보안 구현 환경 정의 : 기밀성·무결성·가용성이 보장된 안전하고 신뢰성 있는 시스템 보안을 구현하기 위한 조건·장소·여건. 관리적·기술적·물리적 차원으로 구분.
- 관리적 보안 구현 환경 : 시스템 보안 정책, 보안 규정 및 절차서, 보안 조직 및 인원 등.
- 기술적 보안 구현 환경 : 시스템 보안 솔루션 및 패키지 등.
- 물리적 보안 구현 환경 : IDC, 보안 통제 공간, 잠금장치, 출입 통제 장비 등.

#### 시스템 보안 구현 계획 개요
- 시스템 보안 구현 계획 정의 : 보안 설계 내용대로 현장에서 구현하기 위한 세부 범위·일정·장소·인원·도구·매뉴얼 및 절차서를 포함하여 실행 방안을 구체화하는 작업.
- 포함될 사항 : 범위(추진 범위·세부 업무), 일정(월간·주간·일 단위), 장소(설계·구현·테스트 장소), 인원(범위·일정별 작업자), 기타 준비 사항(매뉴얼·절차서·도구).

#### 시스템 보안 구현 도구
- 도구 정의 : 어떤 일을 할 때 쓰는 연장·수단·방법. 시스템 보안 구현 도구는 시스템의 기밀성·무결성·가용성을 보장하고 취약점으로부터 노출되지 않도록 구현하기 위해 필요한 수단·방법.
- MBSA(Microsoft Baseline Security Analyzer) : Windows 시스템에서 틀리기 쉬운 보안 관련 설정을 간단히 확인하는 도구.
- Nmap : 서버 관리자가 자체 시스템 스캔을 통해 자신이 모르는 사이에 다른 포트가 열려 있는지 등을 확인하는 도구.
- OWASP ZAP : OWASP에서 제공하는 웹 취약점 점검 도구로 OWASP Top10 주요 웹 취약점을 점검.

#### 시스템 보안 구현 준비 보강 개념
- IDC(Internet Data Center) : 시스템 보안 구현을 위해 확보하는 안전하고 신뢰성 있는 물리적 장소로, 통상 IDC 혹은 별도 통제된 전산실 등의 공간을 확보한다.
- WBS(Work Breakdown Structure) : 시스템 보안 구현의 추진 목표와 구현 범위에 따라 작성하는 세부적인 업무 수행 내역.

## 학습 2. 시스템 보안 구현하기

### 2-1. 시스템 보안 구현

#### 시스템 보안 구현
- 시스템 보안 구현 개념 : 안전하고 신뢰성 있는 하드웨어·소프트웨어로 구성된 시스템 보안이 작동하도록 서버·소프트웨어 구성을 변경하거나 각종 파라미터를 설정하는 것.
- 시스템 보안 구현 범위 : 하드웨어 서버·소프트웨어의 계정 관리, 패스워드 관리, 접속 관리, 세션 관리, 권한 관리, 로그 관리, 취약점 관리 등.

#### 시스템 인증
- 시스템 인증 개념 : 참이라는 근거가 있는 무언가를 확인·확증하는 행위. 컴퓨터 시스템 보안에서 인증은 로그인 요청 등으로 통신상에서 보내는 사람의 디지털 정체성을 확인하는 과정.
- 시스템 인증의 4가지 방법 :
  - 알고 있는 것(Something You Know) : 머릿속에 기억하는 정보로 인증(예: 패스워드).
  - 가지고 있는 것(Something You Have) : 신분증·OTP 장치 등으로 인증(예: 출입카드).
  - 스스로의 모습(Something You Are) : 홍채·지문 등 생체 정보로 인증.
  - 위치하는 곳(Somewhere You Are) : 접속 위치의 적절성 확인(예: 콜백, GPS).

#### 접근 통제 정책 종류
- 임의적 접근 통제 정책(DAC) : 객체에 접근하려는 주체의 접근권한에 따라 접근을 통제. 사용자가 권한을 임의로 추가·철회 가능, 데이터 소유자가 신분에 따라 임의 제어. ACL을 사용한 신원·사용자 기반 통제.
- 강제적 접근 통제 정책(MAC) : 객체에 포함된 정보의 비밀성과 그에 대한 주체의 권한에 근거하여 접근을 제한. 최상위 보안등급자도 모든 자료 열람 불가. 보안등급·규칙·관리 기반 통제.
- 역할 기반 접근 통제 정책(RBAC) : 임의적 접근 통제의 단점을 보완(비임의적 접근 통제). 사용자에게 최소 권한을 부여해 남용을 방지하고, 직무에 따라 허가를 결정하여 직원 변경이 잦은 구조에 유리.
- 접근 통제 행렬 : 주체를 열, 객체를 행에 표시하여 권한을 부여. 주체·객체가 커지면 행렬 유지가 어려움.
- 접근제어(AC) : 사용자에 따라 공유 데이터 접근 권한을 제한. 접근 허용을 결정하는 인증과, 사용자 행위를 강제 제약하는 인가가 있음.

#### 계정 관리 구현
- 루트(root) 계정 : 컴퓨터에서 전체적으로 자원을 관리할 수 있는 총괄 권한을 가진 유일한 특별계정. 원격에서 루트 계정에 직접 로그인을 허용하면 불법 침입자가 로그인하여 시스템 보안이 침해될 수 있으므로 원격 접속을 제한할 필요가 있다.
- SSH(Secure SHell) : 보안을 위해 서버 OS가 기본적으로 Telnet 접속을 허용하지 않으므로 사용하는 안전한 원격 접속 방식. 루트 계정 원격 접속 제한을 위해 sshd_config 파일의 PermitRootLogin 파라미터를 no로 설정한다.
- 불필요한 계정 제거 : 운영체제(OS)나 패키지 설치 시 기본 생성되는 default 계정은 대부분 default 패스워드를 사용해 패스워드 추측 공격에 악용될 수 있으므로, 이용하지 않는 default 계정 및 의심스러운 특이한 계정의 존재 유무를 확인한 후 삭제하는 것.
- 관리자 그룹 최소 계정 등록 : 루트(root) 계정이 속한 그룹은 시스템 운영 파일에 대한 접근권한이 부여되어 있으므로 최소한의 계정만 등록되어 있어야 하며, 등록된 불필요한 계정을 삭제하는 것.

#### 패스워드 관리 구현
- 셰도 패스워드(shadow password) : /etc/passwd 파일은 누구나 읽기가 가능해 일방향 암호화된 패스워드 정보가 노출될 수 있으므로, 패스워드를 특정 장소에 암호화하여 저장해 보호하는 기술.
- DES 일방향 암호화 : 사용자 계정 패스워드를 일방향 암호화(DES)하여 /etc/passwd 파일에 계정 정보와 함께 저장하는 방식. DES 알고리즘은 보안성이 약해 평문으로 해독될 수 있는 취약점이 있다.
- 패스워드 무작위 대입 공격(Brute Force Attack) : 외부 침입자가 가능한 패스워드를 무작위로 대입하여 패스워드를 알아내려는 공격. 패스워드 입력 실패 횟수 제한, 최소 길이 설정 등으로 대비한다.
- 패스워드 추측 공격(Password Guessing) : 외부 침입자가 패스워드를 추측하여 알아내려는 공격. 패스워드 입력 실패 횟수 제한, 최소 길이 설정 등으로 대비한다.
- 계정 잠금 임계값 : 패스워드 입력 실패 횟수를 적정하게 제한하여 일정 횟수를 초과하면 계정을 잠금으로써 패스워드 유출 위험을 줄이는 설정.
- 패스워드 최대 사용 기간 : 악의적인 사용자로부터 계속적인 접속을 차단하기 위해 패스워드의 최대 사용 기간을 설정하여 주기적으로 패스워드를 변경하도록 하는 설정.
- 패스워드 최소 사용 기간 : 최소 사용 기간을 설정하지 않으면 사용자가 익숙한 패스워드로 변경·재사용하여 정기적 변경이 무의미해질 수 있으므로, 이를 방지하기 위해 설정하는 패스워드 변경 후 최소 유지 기간.

#### 파일시스템 접근제어·권한 구현
- 세션 유지 시간(timeout) : 계정이 접속된 상태로 방치되면 권한 없는 사용자에게 중요 시스템이 노출될 수 있으므로, 일정 시간 이후 어떠한 이벤트도 발생하지 않으면 연결을 종료하도록 설정하는 값.
- NFS(Network File System) : 서버의 로컬디스크 영역을 네트워크로 공유하는 서비스로, 파일 공유를 통한 보안 위험이 존재하므로 사용하지 않는 것이 보안에 안전하다.
- Apache 상위 디렉터리 접근 금지 : 상위 경로로 이동이 가능하면 하위 경로에서 상위 경로로 이동해 해킹당할 위험이 있고 유니코드 버그(Unicode Bug) 및 서비스 거부 공격에 취약해질 수 있으므로, 상위 경로 이동 문자 사용을 제한하고 디렉터리 내 파일 호출 시 사용자 인증을 수행하도록 설정하는 것.
- /etc/passwd 파일 : 사용자의 ID, 패스워드, UID, GID, 홈 디렉터리, 셸 정보를 담고 있는 중요 파일. 관리자 이외 사용자가 접근 시 루트(root) 권한 획득이 불가능하도록 접근을 제한해야 한다.
- /etc/shadow 파일 : 시스템에 등록된 모든 계정의 패스워드를 암호화된 형태로 저장·관리하는 중요한 파일. ID 및 패스워드 정보가 외부로 노출되지 않도록 루트(root) 계정을 제외한 모든 사용자의 접근을 제한해야 한다.
- /etc/services 파일 : 서비스 관리를 위해 사용되는 파일. 일반 사용자가 접근·변경하면 정상 서비스를 제한하거나 허용되지 않은 서비스를 악의적으로 실행시켜 침해사고를 발생시킬 수 있다.
- Cron : cron.allow 파일과 cron.deny 파일을 통해 명령어 사용자를 제한할 수 있는 시스템. 접근제한 파일의 권한이 잘못되면 권한 획득자가 임의 계정을 등록해 불법 예약 파일 실행으로 시스템 피해를 일으킬 수 있다.

#### DBMS 보안 관리 구현
- DBMS 보안 관리 : 기업·조직의 중요 데이터를 저장·관리하는 DBMS에 대해 계정 관리, 접근 관리를 수행하고 DBMS 벤더가 제공하는 최신 보안 패치를 적용하는 것.
- DBMS 디폴트(Default) 계정 관리 : DBMS 설치 시 기본 생성되는 디폴트 계정을 변경하지 않고 사용하면 공격자가 기본 계정 정보로 침투해 데이터를 탈취·위변조할 수 있으므로, 기본 생성 계정의 패스워드 및 권한을 변경하는 것.
- DBMS 원격 접속 제한 : DBMS가 외부 인터넷망에서 직접 접속되면 중요 데이터가 탈취·위변조될 위협이 있으므로, 외부에서 접속할 수 없는 내부 네트워크에 설치하고 주요 관리자 계정은 원격 접속을 허용하지 않으며 응용시스템 접속 계정은 IP 필터링 등으로 통제하는 것.
- 응용시스템(WEB, WAS) 접속 계정 최소 권한 : 웹 응용시스템에서 DBMS를 사용하는 계정에 DBA 등 과도한 권한이 설정되면 웹서버를 통해 DBMS로 침투당할 수 있으므로, 데이터 삽입·삭제·업데이트 등 최소한의 권한만 설정하는 것.

#### 웹 취약점
- 웹 취약점 : 웹서비스를 이용하여 서버 내 중요 자원 탈취, 변조, 명령어 실행 등의 공격에 이용될 수 있는 보안 취약점.
- 운영체제 명령어 삽입 : URL의 파라미터 등 사용자 입력값에 의해 시스템 명령어를 삽입해 요청하는 경우 적절한 검증이 없으면 서버 내 명령어로 실행되는 취약점.
- SQL 인젝션 : URL 및 사용자 입력값에서 적절한 검증이 없을 경우 SQL문 구성에 사용되어 공격자가 의도한 SQL문으로 실행되는 취약점.
- 크로스 사이트 스크립트(XSS) : 홈페이지 게시판, 이메일 등으로 악성 스크립트를 전파해 사용자 브라우저에서 악성 스크립트가 강제 실행되는 취약점.
- 불충분한 인증 및 인가 : 사용자 인증 및 접근제어가 미흡하여 비인가자 접근, 허용되지 않은 불법 접근 등을 할 수 있는 취약점.
- 취약한 패스워드 복구 : 취약한 비밀번호 설정으로 비밀번호 복구 기술에 의해 비밀번호를 탈취·변경할 수 있는 취약점.
- 파일 업로드 : 악의적인 스크립트가 포함된 파일을 업로드하여 서버에서 실행해 중요 정보를 탈취할 수 있는 취약점.
- 데이터 평문 전송 : 서버와 사용자 브라우저 간의 구간 암호화가 되지 않은 상태로 통신 시 주요 정보 등이 평문으로 노출되는 취약점.
- URL 파라미터 변조 : 인증 및 사용자 검증을 거치지 않고 파라미터 조작을 통해 게시물 조작이나 주요 정보 탈취가 가능한 취약점.

### 2-2. 시스템 보안 구현 검증

#### 테스트 개념
- 테스트 정의 : 소프트웨어 제품·서비스의 품질에 대한 정보를 이해관계자에게 제공하기 위해 수행된 평가.
- 테스트 목적 : 소프트웨어 결함을 탐지·수정하는 것. 모든 조건에서 정상 작동함을 입증할 수는 없고 특정 조건에서 작동하지 않는 경우를 발견. 요구사항 충족·정확한 응답·수용 가능 시간·사용성·설치 실행·기대 결과 달성 여부를 평가.
- 테스트 범위 : 계정 관리, 패스워드 관리, 접속 관리, 권한 관리, 로깅 관리, 취약점 관리 등.

#### 결함 관리
- 결함의 개념 : 소프트웨어가 바르게 동작하지 않는 원인이 되는 프로그램상의 오류. 요구 명세·설계의 오류도 결함(버그)에 포함.
- 결함 관리의 개념 : 결함 관리 활동으로 발생한 결함의 재발생을 막고 유사 결함 발견 시 처리 시간을 단축하기 위한 활동.
- 결함 관리의 절차 : 에러 발견 → 에러 등록(결함 관리 대장) → 에러 분석(단순 에러/실제 결함) → 결함 확정 → 결함 할당(담당자 지정) → 결함 조치(수정) → 결함 조치 검토 및 승인(확인 테스트 후 조치 완료).

#### 시스템 보안 테스트 개요
- 시스템 보안 테스트 개념 : 시스템을 유지하는 데이터·구조를 보호하고 사용자가 원하는 대로 기능이 유지되는지 확인하는 것.
- 시스템 보안 테스트 대상 : 계정·패스워드 관리, 세션 관리, 접근제어, 권한 관리, 로그 관리, 취약점 관리 (각 개념은 1-1과 동일).

#### 결함 결과 관리
- 결함의 분야별·심각도별 목록화 : 시스템 보안 테스트 결과 발견된 결함을 계정·패스워드·세션·접근제어·권한·로그·취약점 관리 등 분야별로, 심각도에 따라 상·중·하로 구분하여 취합 정리하는 것.
- 시스템 보안 결함 관리 대장 : 테스트 결과 발견된 결함을 관리 번호, 결함 내역, 분야, 결함 발견자, 결함 해결 담당자, 결함 원인, 결함 해결 완료일, 비고 등으로 구분하여 작성·기록하는 문서.
