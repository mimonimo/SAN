# 시스템 보안 구축 — 개념 정리

> 학습모듈 목표: 시스템 보안요구사항을 정의하고 안전한 정보시스템을 설계·구현·검증할 수 있다.

---

## 학습 1. 시스템 보안 설계하기

### 1-1. 시스템 보안요구사항 정의

#### 시스템 보안 개념
- 시스템 보안의 정의 : 시스템 구성 요소·자원의 기밀성·무결성·가용성을 보장하기 위해 취해지는 활동. 외부 피해(해커·바이러스·자연재해)와 내부 피해(불만·부정직·해고당한 종업원)로부터 조직의 컴퓨터 시스템·데이터베이스·네트워크 등 정보 자원을 보호하는 방법.

#### 시스템 보안 범위
- 개념 : 시스템 보안은 주로 계정·패스워드 관리, 세션 관리, 접근제어, 권한 관리, 로그 관리 및 취약점 관리, DBMS 관리 등을 포함한다.

**1. 계정** : 적절한 권한을 가진 사용자를 식별하기 위한 가장 기본적인 인증 수단. 계정(account)이란 한 사용자에게 소유되는 모든 파일·자원·정보를 말함. 계정과 패스워드 관리가 보안의 시작.

**2. 패스워드**
- 패스워드의 정의 : 허가 없이 DB·파일에 접속하는 것을 방지하기 위한 안전 대책으로, 사용자 ID와 함께 입력하여 정당한 사용자임을 식별하도록 주컴퓨터에 전달하는 고유의 문자열.
- 패스워드의 기능 : 온라인 정보 서비스는 패스워드가 없으면 이용 불가. 주컴퓨터가 입력된 ID·패스워드를 등록된 것과 대조하여 일치 시 접속 허용.

**3. 세션 관리**
- 세션의 개념 : 망 환경에서 사용자/컴퓨터 간 대화를 위한 논리적 연결, 또는 프로세스 간 통신 시 메시지 교환으로 서로를 인식한 이후부터 통신을 마칠 때까지의 기간.
- 세션 관리의 개념 : 활성화된 접속에 대한 관리로, 일정 시간 경과 시 세션을 종료하고 비인가자의 세션 가로채기를 통제.

**4. 접근제어**
- 접근의 개념 : 데이터가 있는 곳을 알아내어 가져오는 것, 또는 사용자가 시스템·프로그램을 이용하기 위해 최초로 접속을 시도하는 것.
- 접근제어의 개념 : 시스템이 다른 시스템으로부터 적절히 보호되고 꼭 필요한 사람에 의해서만 필요한 서비스를 제공하도록 전체 시스템 차원에서 접근을 통제하는 것.

**5. 권한 관리**
- 권한의 개념 : 직책 수행에 필요한 결정권 등을 합한 것. 조직 내 위치로 인해 갖는 공식적인 힘이며 합법성이 강조됨. 한 개인·집단을 지배할 수 있는 권리(right).
- 권한 관리의 개념 : 각 사용자가 적절한 권한으로 적절한 정보 자산에 접근하도록 통제하는 것.

**6. 로그 관리**
- 로그의 개념 : 데이터를 기록한 것 또는 데이터의 인쇄된 기록.
- 로그 관리의 목적 : 시스템 내부·외부에서 시스템에 영향을 미칠 경우 해당 사항을 기록하여 문제 해결·예방 활동에 활용.

**7. 취약점 관리**
- 취약점 개념 : 공격자가 시스템의 정보 보안을 낮추는 데 사용되는 약점. 시스템 민감성·결함, 결함에 대한 접근, 결함에 대한 공격 가능성을 의미.
- 취약점 관리의 개념 : 취약점을 확인·분류·치료·완화하는 주기적인 과정.
- 취약점 관리의 목적 : 계정·접근제어 등이 잘 갖춰져도 시스템 자체의 결함으로 보안 문제가 발생할 수 있으며, 이런 결함을 체계적으로 관리하는 것.

**8. DBMS 관리**
- DBMS 개념 : 기업·조직이 보유하는 데이터베이스를 저장·관리하는 시스템(Data Base Management System). 전사적 자원·서비스 데이터를 포함하므로 보안이 중요.
- DBMS 관리의 목적 : DBMS 계정 관리, DB 접근 권한, DB 암호화, DB 백업 등을 관리.

### 1-2. 시스템 보안 설계

#### 시스템 보안 설계의 개념
- 시스템 보안 설계 개념 : 악성코드·개인정보 유출·중요 정보 노출·비밀 데이터 손실 등의 비즈니스 문제를 방지·해결하기 위해 안전한 정보시스템 환경을 구축하고자 관리적·기술적·물리적 보안 차원에서 보안 구축 계획을 수립하고 보안 설계서를 작성하는 것.

**시스템 보안 설계 대상** : 계정 관리, 패스워드 관리, 세션 관리, 접근 통제, 권한 관리, 취약점 관리(각 개념은 1-1과 동일).

**시스템 보안 설계 원칙**
- 원칙 1 : 자원에 적용되는 보안 수준은 조직에 주는 가치에 비추어 적절해야 하며, 보안 비용은 그 혜택에 비추어 합리적이어야 한다.
- 원칙 2 : 보안 아키텍처는 변화하는 보안 필요·요구사항을 수용하고, 보호 레벨 변화 시에도 기본 아키텍처 수정 없이 지원하며, 미래 확장을 수용할 만큼 확장성이 있어야 한다.
- 원칙 3 : 조직이 안전한 업무를 전자적으로 수행하도록 통합된 보안 서비스를 제공해야 한다.
- 원칙 4 : 모든 컴퓨터 플랫폼에 걸쳐 일관성 있는 프레임워크를 제공해야 한다.

### 1-3. 시스템 보안 구현 준비

#### 시스템 보안 구현을 위한 환경 개요
- 시스템 보안 구현 환경 정의 : 기밀성·무결성·가용성이 보장된 안전하고 신뢰성 있는 시스템 보안을 구현하기 위한 조건·장소·여건. 관리적·기술적·물리적 차원으로 구분.
- 관리적 보안 구현 환경 : 시스템 보안 정책, 보안 규정 및 절차서, 보안 조직 및 인원 등.
- 기술적 보안 구현 환경 : 시스템 보안 솔루션 및 패키지 등.
- 물리적 보안 구현 환경 : IDC, 보안 통제 공간, 잠금장치, 출입 통제 장비 등.

#### 시스템 보안 구현 계획 개요
- 시스템 보안 구현 계획 정의 : 보안 설계 내용대로 현장에서 구현하기 위한 세부 범위·일정·장소·인원·도구·매뉴얼 및 절차서를 포함하여 실행 방안을 구체화하는 작업.
- 포함될 사항 : 범위(추진 범위·세부 업무), 일정(월간·주간·일 단위), 장소(설계·구현·테스트 장소), 인원(범위·일정별 작업자), 기타 준비 사항(매뉴얼·절차서·도구).

#### 시스템 보안 구현 도구
- 도구 정의 : 어떤 일을 할 때 쓰는 연장·수단·방법. 시스템 보안 구현 도구는 시스템의 기밀성·무결성·가용성을 보장하고 취약점으로부터 노출되지 않도록 구현하기 위해 필요한 수단·방법.
- MBSA(Microsoft Baseline Security Analyzer) : Windows 시스템에서 틀리기 쉬운 보안 관련 설정을 간단히 확인하는 도구.
- Nmap : 서버 관리자가 자체 시스템 스캔을 통해 자신이 모르는 사이에 다른 포트가 열려 있는지 등을 확인하는 도구.
- OWASP ZAP : OWASP에서 제공하는 웹 취약점 점검 도구로 OWASP Top10 주요 웹 취약점을 점검.

## 학습 2. 시스템 보안 구현하기

### 2-1. 시스템 보안 구현

#### 시스템 보안 구현
- 시스템 보안 구현 개념 : 안전하고 신뢰성 있는 하드웨어·소프트웨어로 구성된 시스템 보안이 작동하도록 서버·소프트웨어 구성을 변경하거나 각종 파라미터를 설정하는 것.
- 시스템 보안 구현 범위 : 하드웨어 서버·소프트웨어의 계정 관리, 패스워드 관리, 접속 관리, 세션 관리, 권한 관리, 로그 관리, 취약점 관리 등.

#### 시스템 인증
- 시스템 인증 개념 : 참이라는 근거가 있는 무언가를 확인·확증하는 행위. 컴퓨터 시스템 보안에서 인증은 로그인 요청 등으로 통신상에서 보내는 사람의 디지털 정체성을 확인하는 과정.
- 시스템 인증의 4가지 방법 :
  - 알고 있는 것(Something You Know) : 머릿속에 기억하는 정보로 인증(예: 패스워드).
  - 가지고 있는 것(Something You Have) : 신분증·OTP 장치 등으로 인증(예: 출입카드).
  - 스스로의 모습(Something You Are) : 홍채·지문 등 생체 정보로 인증.
  - 위치하는 곳(Somewhere You Are) : 접속 위치의 적절성 확인(예: 콜백, GPS).

#### 접근 통제 정책 종류
- 임의적 접근 통제 정책(DAC) : 객체에 접근하려는 주체의 접근권한에 따라 접근을 통제. 사용자가 권한을 임의로 추가·철회 가능, 데이터 소유자가 신분에 따라 임의 제어. ACL을 사용한 신원·사용자 기반 통제.
- 강제적 접근 통제 정책(MAC) : 객체에 포함된 정보의 비밀성과 그에 대한 주체의 권한에 근거하여 접근을 제한. 최상위 보안등급자도 모든 자료 열람 불가. 보안등급·규칙·관리 기반 통제.
- 역할 기반 접근 통제 정책(RBAC) : 임의적 접근 통제의 단점을 보완(비임의적 접근 통제). 사용자에게 최소 권한을 부여해 남용을 방지하고, 직무에 따라 허가를 결정하여 직원 변경이 잦은 구조에 유리.
- 접근 통제 행렬 : 주체를 열, 객체를 행에 표시하여 권한을 부여. 주체·객체가 커지면 행렬 유지가 어려움.
- 접근제어(AC) : 사용자에 따라 공유 데이터 접근 권한을 제한. 접근 허용을 결정하는 인증과, 사용자 행위를 강제 제약하는 인가가 있음.

### 2-2. 시스템 보안 구현 검증

#### 테스트 개념
- 테스트 정의 : 소프트웨어 제품·서비스의 품질에 대한 정보를 이해관계자에게 제공하기 위해 수행된 평가.
- 테스트 목적 : 소프트웨어 결함을 탐지·수정하는 것. 모든 조건에서 정상 작동함을 입증할 수는 없고 특정 조건에서 작동하지 않는 경우를 발견. 요구사항 충족·정확한 응답·수용 가능 시간·사용성·설치 실행·기대 결과 달성 여부를 평가.
- 테스트 범위 : 계정 관리, 패스워드 관리, 접속 관리, 권한 관리, 로깅 관리, 취약점 관리 등.

#### 결함 관리
- 결함의 개념 : 소프트웨어가 바르게 동작하지 않는 원인이 되는 프로그램상의 오류. 요구 명세·설계의 오류도 결함(버그)에 포함.
- 결함 관리의 개념 : 결함 관리 활동으로 발생한 결함의 재발생을 막고 유사 결함 발견 시 처리 시간을 단축하기 위한 활동.
- 결함 관리의 절차 : 에러 발견 → 에러 등록(결함 관리 대장) → 에러 분석(단순 에러/실제 결함) → 결함 확정 → 결함 할당(담당자 지정) → 결함 조치(수정) → 결함 조치 검토 및 승인(확인 테스트 후 조치 완료).

#### 시스템 보안 테스트 개요
- 시스템 보안 테스트 개념 : 시스템을 유지하는 데이터·구조를 보호하고 사용자가 원하는 대로 기능이 유지되는지 확인하는 것.
- 시스템 보안 테스트 대상 : 계정·패스워드 관리, 세션 관리, 접근제어, 권한 관리, 로그 관리, 취약점 관리 (각 개념은 1-1과 동일).
