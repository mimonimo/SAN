# SW개발 보안 구축 — 개념 정리

> 학습모듈 목표: 정의된 보안요구사항에 따라 응용프로그램 보안요구사항을 명세·설계하고, 보안약점이 제거된 SW를 구현·테스트할 수 있다.

---

## 학습 1. SW개발 보안 설계하기

### 1-1. SW개발 보안요구사항 명세와 설계

#### SW개발 보안의 개요
- 개념 : SW개발 보안은 사이버 보안 공격 위협에 대응하도록 SW개발 단계부터 보안약점을 제거하여 안전한 SW를 구축하는 활동. 해킹 등 사이버공격의 원인인 보안 취약점을 개발 단계에서 미리 제거하고, SDLC 단계별로 보안 업무를 수행해 안전한 SW를 개발·운영하기 위한 개발 방법.
- SW개발 보안의 3요소 : 정보 보안 3요소인 기밀성(confidentiality)·무결성(integrity)·가용성(availability)을 지키고, 서버 취약점을 사전에 방지하여 외부·내부 위협으로부터 위험을 최소화하는 구축 방법.

**보안의 위험 요소**
- 자산(Asset) : 조직이 보유한 정보화 자산 및 데이터(서버·네트워크·소프트웨어·중요 데이터 등).
- 위협원(Threat agents) : 자산에 피해를 줄 수 있는 내·외부 공격 위험 주체(비인가 직원·외부 해커·자연재해 등).
- 위협(Threat) : 자산에 피해를 주는 위협원의 공격 행동(해킹·위변조·삭제·불법 유통 등).
- 취약점(Vulnerability) : 위협이 발생할 수 있는 상황(Injection·웹 취약점·취약한 암호화·미사용 포트 오픈 등).
- 위험(Risk) : 위협원이 취약점을 이용하여 자산에 위해를 입힐 수 있는 상황 예측.

**SW 보안약점과 취약점의 개념**
- 보안약점(Weakness) : 개발 단계에서 발생하는 보안 관련 오류(정적 오류). 공격에 활용될 여지가 있는 이론상 위험 요소. 분류 체계는 CWE(Common Weakness Enumeration).
- 취약점(Vulnerability) : 운영 단계에서 발생하는 보안 관련 오류(동적 오류). 실제로 공격 구현이 가능한 요소. 분류 체계는 CVE(Common Vulnerabilities and Exposures). 보안약점이 존재하면 취약점으로 발전할 가능성이 높음.

#### SW개발 보안 방법론
- 개념 : 기존 SW개발 생명주기(SDLC)에서 안전한 SW 개발을 위해 단계별로 요구되는 보안 활동을 적용한 방법론.
- SDLC 단계별 보안 활동 :
  - 요구사항 분석 : 보안 요구 항목·시스템 보안 항목 식별, 개발 환경 보안 프레임워크·운영 보안 환경 분석 → 산출물: 보안요구사항 정의서, 보안적용계획서.
  - 설계 : 보안 요구 항목 구현 방안 설계, 보안 프레임워크 인증·인가 설정, 화면 설계 등 → 산출물: 보안적용계획서, 보안 프레임워크.
  - 구현 : 시큐어 코딩(secure coding), 소스코드 보안약점 진단·조치, 코드 리뷰 → 산출물: 보안약점 진단 결과서, 보안약점이 제거된 소스코드.
  - 테스트 : 오류 유발 테스트 케이스 기반 테스트, 공격 시나리오 테스트, 모의 침투 테스트 → 산출물: 테스트 결과서, 취약점 조치 결과서.
  - 운영 및 유지보수 : 지속적 보안 패치, 보안약점·취약점 진단 및 조치 → 산출물: 보안 패치 결과서, 진단·조치 결과서.

#### SW개발 보안의 요구공학 프로세스
- 개념 : 보안 요구공학은 크게 보안요구사항 개발과 보안요구사항 관리로 나뉜다.
- 요구사항 도출 : 법·제도 준거성(Compliance), 내·외부 시스템 환경, 시스템 보안 요구사항을 식별·도출하고 이해관계자 협의로 수집·정제·분류(제안서·계약서·과업지시서·회의록 등 참조).
- 요구사항 분석 : 분류된 요구사항을 토대로, 구현 난이도·비용·인력 등 효율성을 고려해 제약조건을 판별하고 필요 시 대안을 제시.
- 요구사항 명세 : 분석서를 통해 요구사항 정의서 도출. SW 시스템의 목표 기술과 기능·비기능 요구사항을 명세.
- 요구사항 확인과 검증 : 이해관계자·조직 성숙도·표준을 고려하여 문제 보고서를 작성하고 경영진 승인으로 확정. 구현 가능성·테스트 가능성·추적성을 검토.
- 요구사항 관리 : 환경 변화·시간 흐름에 따라 보안요구사항이 변경될 수 있으므로 지속 관리·갱신. 정식 승인 후 변경, 보안요구사항 추적 매트릭스로 산출물·테스트 시나리오를 관리.

> 의사소통 관리: 정형적·서면(복잡한 이해 상충 해결), 정형적·구두(즉석 피드백, 발표), 비정형적·서면(단순 정보 요청, 메모·이메일), 비정형적·구두(개인 상담, 회의).

#### SW개발 보안의 설계
- 개념 : 운영·개발 애플리케이션이 조직 사업 방향에 맞는 보안 목적을 정의하고, 보안 위험 분석 및 전사 보안 계획을 작성하는 것.
- 요구사항 반영 : 분석 단계 보안요구사항 분석서를 반영하여 감지하지 못하는 보안 위험·통제를 고려해 보안 설계서(기본·상세 설계)를 작성.
- 요구사항 설계 반영 : 조직 표준(기술·비즈니스 요건) 적합성 확인, 실제 구현 가능성 판단, 보안요구사항을 설계에 적절히 반영.
- 환경 설계 반영 : 외부 공격으로부터 개발/운영 환경 보호를 위한 네트워크 보안 구성, 개발/운영 네트워크 분리, DMZ 구간 접근·계정 관리 점검.
- 보안적용계획서 활용 : 보안요구사항 명세서 기반으로 설계 단계 보안적용계획서를 작성해 아키텍처정의서·화면설계서·보안 프레임워크 설계에 활용.

### 1-2. SW개발 보안 환경 구축 일정 계획 수립

#### 보안성이 강화된 응용프로그램 구현을 위한 환경의 구축
- 개념 : 대상 시스템을 개발 환경에 배포하는 데 쓰이는 취약점 진단 도구, 모니터링·결함 추적/관리 시스템 등으로 구성.
- 보안 취약점 환경 구축 기능 : 대상 시스템을 테스트 환경에 배포, 진단 도구·모니터링 툴 설치 관리, 환경 설정 정보 문서화.
  - 진단 서버 : 취약점 진단에 필요한 CPU·메모리 등이 장착된 서버.
  - 스토리지 : 취약점 진단 데이터를 담는 저장 공간.
  - 네트워크 : 서버와 클라이언트를 연결하는 연결망(내부는 LAN으로 구성).
- 취약점 모니터링 환경 기능 : 진단 진행 중 인프라·대상 시스템 모니터링 수행 및 모니터링 정보 저장.

#### 보안성이 강화된 응용프로그램 구현을 위한 일정 계획의 수립
- 개념 : 발주·계약 이전에 개발 전략·계획을 수립하고 조직의 보안 정책을 일정에 반영. 역할과 책임을 명시.
- 보안 전문가 선정 : 모든 개발 과정을 경험 있는 보안 전문가·실무자가 수행하도록 함.
- 보안 범주 식별 : 사고의 잠재적 파급 효과를 기반으로 조직 보안요구사항을 도출·선택해 일정에 반영.
- 보안요구사항 분석 : 개발자·공급자 선택, 부서 간 역할·책임, 전사 프로젝트 관리 등을 기술.
- 독립적 테스트 계획 기획 : 보안 강화 여부를 직간접 확인하기 위한 취약점 점검 계획 기술.

#### 분석·설계 보안 항목과 구현 단계 보안약점의 관계
- 개념 : 안전한 SW를 위해 분석·설계 단계부터 보안성을 고려하고, 구현 단계에서는 시큐어 코딩으로 보안약점이 없도록 함. (예) DBMS 조회·결과 검증↔SQL Injection, 웹서비스 요청·결과 조회↔XSS, 웹 기능 요청 유효성 검증↔CSRF, 허용 범위 메모리 접근↔메모리 버퍼 오버플로우, 비밀번호 관리↔비밀번호 평문 기록, 암호화 키 관리↔하드코딩된 암호화 키 등.

## 학습 2. SW개발 보안 구현하기

### 2-1. SW개발 보안 구현

#### SW개발 보안 구현을 위한 시큐어 코딩
- 개념 : 안전한 SW를 개발해 보안 위협을 예방·대응하고, 개발 시 보안성을 고려하여 SW 보안약점을 사전에 제거하기 위해 사용하는 코딩.
- 시큐어 코딩의 목적 : ① 보안 취약점·결함 방지(SQL Injection·Zero Day·보안 장비 우회 등 사전 제거), ② 안전한 대고객 서비스 확대, ③ 안정성·신뢰성 확보(설계 개념·코드 수준 대응).
- 시큐어 코딩 적용 방안 : 보안 정책 수립, 시큐어 코딩 가이드 작성, 가이드 교육, 시큐어 코딩 자동화 도구(보안약점 진단 도구) 활용. (사후 수정 비용은 시큐어 코딩 비용의 수십 배)
- 소스코드 난독화(obfuscation)의 필요성 : 컴파일·빌드된 실행 코드는 디컴파일 등 역공학으로 소스코드 복원이 가능. 특히 JAVA·C#는 중간 계층 바이트코드로 컴파일되어 손쉽게 디컴파일되므로, 중요 정보 유출·소스 변조를 막기 위해 소스코드를 읽기 어렵게 만드는 난독화가 필요.

#### SW보안을 위한 시큐어 코딩의 점검 내용 (보안약점 7대 유형)

**1. 입력데이터 검증 및 표현** — 입력 데이터에 대한 유효성 사전 검증 체계 부재로 발생.
- SQL Injection : 사용자 입력값이 SQL문 조건으로 구성되어 DB 전체 조회·위변조가 가능한 보안약점.
- Code Injection : 외부 입력값이 실행 명령·코드로 입력되어 서버 자원 탈취·공격.
- 경로 조작 및 자원 삽입 : 외부 입력값으로 서버 자원에 접근하여 수정·삭제·정보 노출, 권한 획득.
- XSS(Cross Site Script) : 웹페이지에 스크립트를 포함시켜 브라우저에서 실행, 의도하지 않은 URL 접근·악성 스크립트 실행.
- 파일 업로드 : 서버 측 실행 가능 파일(jsp·asp·cmd 등) 업로드로 시스템 명령 실행·외부 제어.

**2. 보안 기능** — 인증·접근제어·기밀성·암호화·권한 관리 등을 부적절하게 구현하여 발생.
- 적절한 인증 없는 중요 기능 허용 : 인증 없이 비인가 사용자가 중요 정보·자원에 접근.
- 중요 자원의 잘못된 권한 설정 : 중요 보안 자원에 부적절한 읽기/수정 권한 허가.
- 취약한 암호화 알고리즘 : 간단한 인코딩·취약한 암호화 알고리즘 사용.
- 암호화되지 않은 중요 정보 : 중요 정보를 평문으로 전송·저장.
- 하드코드된 중요 정보 : 소스코드·설정 파일에 암호화 키를 하드코딩.
- 부적절한 전자서명 인증 / 부적절한 인증서 유효성 검증 / 부적절한 인가.

**3. 시간 및 상태** — 병렬·다중 프로세스 환경에서 시간·실행 상태가 부적절하게 처리되어 발생.
- 공유자원 동기화 처리(TOCTOU, 검사 시점과 사용 시점) : 공유 자원 동시 접근 문제. 적절한 동기화(synchronized·mutex) 필요.
- 종료되지 않은 반복문 또는 재귀함수 사용 : 종료 조건·귀납 조건 부재로 무한 반복, 시스템 자원 고갈.

**4. 에러처리(exception)** — 예외를 부적절하게 처리하거나 오류 정보를 그대로 노출하여 발생.
- 오류메시지를 통한 중요 정보 노출 : 예외 시 오류 정보의 시스템 중요 정보 노출.
- 오류 상황 대응 부재 : catch에서 아무 처리도 안 해 오류를 모른 채 계속 실행되는 논리적 오류.
- 부적절한 예외 처리 : 예외 상황에 맞는 처리 조건을 검사하지 않아 잘못된 실행 결과.

**5. 코드 오류** — 잘못된 방법의 코딩으로 발생.
- Null Pointer 역참조 : 객체 생성 실패를 유도할 수 있는 보안약점.
- 부적절한 자원 해제 : 자원 사용 후 반환하지 않아 다른 프로세스가 사용하지 못하는 오류.
- 해제된 자원 사용(동적 메모리 참조 오류) : 해제한 메모리를 다시 참조. 해제 후 null 초기화 필요.
- 초기화하지 않은 변수 사용 : C·Pascal 등에서 미초기화 변수가 임의값을 가져 의도하지 않은 결과.

**6. 캡슐화** — 데이터·기능을 충분히 캡슐화하지 않아 내부 데이터 누출이 가능.
- 잘못된 세션에 의한 데이터 정보 노출 : 싱글톤 객체의 멤버필드 사용 최소화 필요.
- 제거되지 않고 남은 디버그 코드 : 배포 전 디버그 코드 제거 필요.
- Public 메소드로부터 반환된 Private 배열 / Private 배열에 Public 데이터 할당 : 배열 참조가 외부에 노출되어 수정·속성 변경 가능.

**7. API 오용** — 의도에 반하는 방법으로 또는 취약한 API를 사용하여 발생.
- 취약한 API 사용 : 보안에 취약한 API·함수 사용 또는 잘못된 방식 사용.
- DNS Lookup에 의존한 보안 결정 : 도메인명에 의존해 보안 결정 시 DNS 캐시 오염으로 공격 발전.

### 2-2. SW개발 보안 테스트와 결함 관리

#### SW개발 보안 테스트
- 결함 등급의 영향도와 긴급도 : 보안 결함 등급 = 복구 우선순위 = 영향도(잠재적 손실의 영향) × 긴급도(해결 시간의 중요성).
- SW개발 보안 테스트의 종류 :
  - 정적 분석 : SW를 실행하지 않고 소스코드 수준으로 보안약점 분석(개발 과정에서 사용). 초기 발견으로 수정 비용 절감, 단 통합·설계·구조 관점 약점 발견은 제한적.
  - 동적 분석 : SW 실행 환경에서 보안약점 분석(시험 단계에서 사용). 소스코드 불필요, 정확도·커버리지 향상, 단 도구 사용자 수준에 영향, 구조 관점 약점 발견 불가.
- 보안 테스트 일반 원칙 : 개발자와 운영자를 분리하여 공정하게 테스트. 계획서에 따른 실시, 적정성 판단, 결과의 신뢰성 확인, 결함 발견 시 피드백.

#### SW개발 보안 테스트에 따른 결함관리
- 개념 : 반복·지속 테스트 후 발생 결함을 제거·재발 방지하여 고품질 SW를 제공하는 활동.
- SW보안 결함의 개요 : 프로그램과 보안명세서의 차이·불일치, 기대 보안 품질과 실제 관찰 결과의 차이.
- SW보안 결함의 종류 : SW보안 결함(정의 특성과 불일치한 모든 행위), 발견된 보안 결함(운영 전 발견), 잠재적 보안 결함(운영 환경에 전달됨), SW의 특이한 고장(잠재 결함이 운영 중 나타난 이상 징후 집합).
- SW보안 결함관리 프레임워크(절차) : 관리계획(테스트·결함 추적·분류 정의) → 결함 기록 → 결함 검토(원인 파악) → 결함 수정(할당·수정) → 결함 재확인 → 상태 추적 모니터링 → 최종 분석·리포팅.
- 단위 테스트 절차 및 산출물 : 보안 테스트 계획서 작성 → 범위 분석·분할 → 테스트 시나리오 작성 → 테스트 케이스 작성 → 테스트 수행(Defect Report) → 테스트 평가(정성·정량) → 결과 보고.
