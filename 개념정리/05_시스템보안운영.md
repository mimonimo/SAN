# 시스템 보안 운영 — 개념 정리

> 학습모듈: 시스템 보안 솔루션을 운영·개선하고, 신규 위협에 대응하며, 솔루션 업데이트를 적용한다.

---

## 학습 1. 시스템 보안 솔루션 운영하기

### 1-1. 시스템 보안 솔루션 요구사항 분석

#### 시스템 보안 운영의 범위
- 서버 보안 : 정보·자원이 저장된 서버의 보안 운영(서버 보안, DB 보안, 세션 로깅 등).
- PC 보안 : 사용자 단말 PC에 대한 보안 운영(통합 PC 보안, 오피스 PC 보안, 시큐어 프린터 등).
- 보안 관리 : 관리적 보안에 활용되는 보안 운영(스토리지 암호화, 디가우저 등).

#### 시스템 보안 운영의 요구사항 수집 대상 및 항목
- 수집 대상 : ① 정보화 사업 관련 법령·훈령·지침·정책, ② 기업의 정보보호 규정 및 보안 정책, ③ 발주 기업 제안서 내 보안 요구사항.
- 수집 항목 : 보안 운영 정책, 통제 영역·통제 항목, 통제 항목별 통제 수준, 정책·목적 달성을 위한 세부 지침.
- 정보 보안 규정 : 외부 누출 금지 영업 비밀·내부 정보에 대해 준수해야 하는 규칙을 정한 문서. 보안 조직 구성·총책임자(대표자) 선정·부서별 담당자 채택으로 엄격한 보안 관리.

#### 시스템 보안 운영상 보안 솔루션의 개념 및 역할
- 보안 솔루션의 개념 : 보안상 발생하는 기술적 위험 요소를 방어하기 위한 솔루션. 기술 레벨에 따라 구성 요소를 선별 적용.
- 보안 솔루션의 역할 : 기업 내 보안 담당 인원이 1~2명으로 운영·보안 겸직 등 인력이 부족하여 보안 솔루션 의존도가 높음. 따라서 효율적 운영을 위해 솔루션에 대한 정확한 이해와 운영 노하우가 필요.

#### 시스템 보안 솔루션별 보안 기능
- 기업 보안 관리(ESM) : 다양한 보안 시스템과 보안 장비를 연동하여 운영하는 통합 보안 관리 시스템.
- 위협 관리 시스템(TMS) : 사이버 공격을 예측하고 판단하여 대응할 수 있는 체계적인 위협 관제 및 대응 시스템.
- 보안 스마트 카드 : 핸드폰이나 PC에 삽입해 전자 결제 시 신원 인증에 사용하는 인증 제품.
- H/W 토큰 : 전자 서명 생성 및 검증 등이 가능한 보안 하드웨어 장치.
- 일회용 비밀번호(OTP) : 로그인 시 새로운 패스워드를 생성하는 보안 시스템.
- 바이러스 백신 : 바이러스·웜 등 악성 프로그램을 탐지·치료하는 프로그램.
- 안티 스파이웨어 : 스파이웨어를 탐지·치료하는 프로그램.
- 스팸 차단 S/W : 스팸 메일을 차단하는 솔루션.
- Secure OS(보안 운영 체제) : 운영 체제의 보안 결함으로 인한 해킹을 방지하기 위해 보안 기능이 추가된 운영 체제.
- PC 보안(제품) : PC 방화벽·암호화·매체 제어 기능 등을 통합한 PC 보안 제품.
- DB 보안 : 허가받지 않은 사용자의 DB 접근을 제한하고 내부자에 의한 DB 유출에 대비해 접근 제어·암호화·감사를 수행하는 솔루션.
- DRM : 디지털 콘텐츠에 대한 저작권을 보호하는 솔루션.
- NAC : 비인가 사용자 및 단말의 내부 네트워크 접근을 제한하는 솔루션.
- SSO : 한 번의 인증만으로 전 시스템을 하나의 시스템처럼 사용할 수 있는 시스템.
- EAM : 정보 자원에 대한 접근 인증과 접근 권한을 관리하는 통합 인증 관리 솔루션.
- IAM : ID/PW를 종합적으로 관리해 주는 역할 기반의 사용자 계정 관리 솔루션.
- 지문 인식 : 개인마다 다른 지문 정보를 이용하여 신분을 확인하는 바이오 인식 기법.
- 정맥 인식 : 손바닥·손목 등의 혈관 패턴을 이용하여 신분을 확인하는 바이오 인식 기법.
- 얼굴 인식 : 카메라로 입력된 얼굴 영상과 DB 얼굴 영상을 비교하여 신분을 확인하는 기법.
- 다중 인식 : 여러 인식 기술을 함께 사용하여 인식 성능을 향상시키고 신뢰도를 높이는 인식 기법.

### 1-2. 시스템 보안 솔루션 운영

#### 시스템 보안 솔루션 운영 업무의 어려움
- 개념 : 다양한 솔루션 운영으로 일관성·체계성·전문성 확보가 어려움. ① 복잡·비일관 정보 조치 어려움, ② 운영 정보의 체계적 관리 어려움, ③ 개별 평가로는 전체 통일된 보안 수준 평가 어려움, ④ 솔루션별 전문 인력 필요하나 인력 부족·비용 발생.

#### 시스템 보안 솔루션 안정적 운영 업무 프로세스 방법론
- 정보보안 프레임워크의 정의 : 기업 업무 연속성을 위한 네트워크·시스템 등 주요 인프라의 위협 요인을 사전 분석·예방하고, 발생 시 적절히 대응하기 위한 정책·프로세스·기술적 요인의 관계를 나타낸 것.
- 정보보안 프레임워크의 구성 : 기술적 보안(시스템 보안)에 관리적 보안·물리적 보안 요소를 함께 적용.
- 프레임워크 필요성 : 관리적 측면(전사 일관 정책·이벤트 연관 관계 파악), 재무적 측면(중복 투입 방지·통합 운영 비용 절감), 성능적 측면(개별 솔루션의 딜레이·로드 한계 극복).
- 정보 보안 운영 프로세스 : ① 정보보호 대상 현황 파악(우선순위 분석) → ② 예방 대응(예방 정량 지표 분석·조치) → ③ 탐지 대응(탐지 지표 책정·매뉴얼 대응) → ④ 업무 정량화(관리 지표 측정·지속 개선).

#### 정보 보안 관리 시스템 인증 (ISO 17799)
- 개념 : 영국 BSI가 제정한 BS7799 기반의 국제 정보 보안 표준이자 인증 체계로 정보 보안 관리의 베스트 프랙티스를 정의.
- Part 1 : 정보 보안 관리 실행 지침(보안 정책·조직·인적 보안 등 10개 분야, 127개 통제 항목).
- Part 2 : ISMS 규격(비즈니스 위험 접근 기반으로 정보 보안을 수립·구현·운영·검토·개선하는 시스템).

#### 시스템 보안 솔루션 운영 절차상 보안 대책
- 원격 운영 보안 대책 : 원격지 정보 시스템 관리는 원칙적으로 금지하되, 필요 시 정보보호 최고 책임자 승인·접속 단말 및 사용자 인증·한시적 접근 권한 부여·전송 구간 암호화(VPN)·접속 단말 보안·지속적 모니터링 대책을 마련.
- DMZ(Demilitarized Zone) : 공개 서버를 내부 네트워크와 분리하여 설치하고 침입 차단 시스템 등 보안 솔루션으로 보호하는 영역.

## 학습 2. 시스템 보안 솔루션 운영 개선하기

### 2-1. 시스템 보안 솔루션 운영 개선 방안 수립

#### 시스템 보안 솔루션 개선 방안 도출 프로세스
- 개념 : 보안 솔루션도 SW 기반이라 허점이 존재하고, 다수 운영 시 인력 전문성 결여로 취약점이 발생하므로 개선 방안 도출이 필요.
- 절차 : ① 시스템 구성 분석 → ② 보안 위협 도출(STRIDE 등 위협 모델링) → ③ 보안 취약점 진단(점검 항목 개발·진단) → ④ 개선 방안 수립(패치 적용·윈백 등).
- STRIDE 위협 유형 : Spoofing(위장), Tampering(변조), Repudiation(부인), Information disclosure(정보 노출), Denial of service(서비스 거부), Elevation of privilege(권한 상승).
- 8단계 리스크 계산 — DREAD : 취약성 5가지에 점수를 할당해 위협 우선순위를 결정.
  - damage potential : 악용 시 피해 정도를 수치화.
  - reproducibility : 공격 재현 난이도를 점수화.
  - exploit-ability : 실제 공격까지 요구되는 시간·노력을 수치화.
  - affected users : 예상 피해 사용자 수 비율을 수치화.
  - discover-ability : 약점 발견 난이도를 수치화.
- 9단계 공격 보안 대책 수립 — 위협 트리(threat tree) : 루트 노드가 최종 공격 목표, 하위 노드는 상위 공격 달성에 필요한 전제 조건. 우선순위 공격에 경감책을 수립해 이전 개발 단계에 반영.

#### 정보 자산의 식별 및 가치 평가
- 자산 식별 : 전자 정보·SW·HW·종이 문서·부대설비 등 모든 정보 자산을 대상으로, 분류 기준을 정의해 중복·누락 최소화(정보·문서·인력·소프트웨어·설비·기타 5~6개로 분류).
- 자산 평가 : 비밀성·무결성·가용성 3가지 측면을 고(3)·중(2)·저(1)로 평가하고, 합계를 3~5등급으로 세분화.

#### 보안 취약성 진단 방법론 및 절차
- Combined Approach 방법론 : ISO/IEC 1335 TR(GMITS)에서 권고하는 보안 취약성 진단 방법론.
- ISO/IEC 27001 : 파일럿 테스트를 활용한 자가 진단으로 보안 수준을 측정하고 점검 우선순위를 결정하는 데 참고하는 국제 보안 표준 규격.
- 시스템 위협·취약점 진단 유형 : 시스템 진단·웹 애플리케이션 진단·모의 해킹의 3가지 형태로 진행.
- 시스템 보안 진단 절차(5단계) : 대상 식별 → 인벤토리 스캐닝 → 취약점 점검 → 상세 취약점 점검 → 보고서 작성.
- 대상 식별 : 외부 접근에 노출되어 침해 위험성이 높은 공개 서버 등 정보 자산과 이를 보호하는 운영 중인 보안 솔루션을 분석 대상으로 선정하는 단계.
- 인벤토리 스캐닝(inventory scanning) : 보안 취약점 진단 대상 서버의 IP address 등 개략적 정보를 확인하는 단계.
- 상세 취약점 점검 : 시스템 파일 취약점 및 보안 솔루션의 설정 오류 등을 이용해 루트 권한 획득 등의 취약점을 점검하는 단계.
- 모의 해킹 진단 : 외부에서 웹 환경을 이용해 내부망 공격을 시도하여 침투 가능성·안전성을 테스트하고, 내부에서는 파일명 변경 등으로 유출 가능성·안정성을 테스트하는 취약점 진단 기법.

#### 위험도 측정 및 위험 처리
- 기준 위험도(C) : 위협 점수(A)에 취약성 양호(1점)를 곱한 값.
- 실제 위험도(D) : 위협 점수(A)에 취약성 점수(B)를 곱한 값으로, 기준 위험도보다 점수가 높을수록 보안 취약성·위험도가 높음.
- 위험 수준 결정(DoA, degree of assurance) : 비용 대비 효율성·조직 상황을 고려해 관련 부서·담당자와 협의하여 수용 가능한 위험 수준을 결정하는 것. 실제 위험도가 DoA를 초과하는 항목에 대해서만 보안 대책을 강구.
- 위험 처리 : 위험도 측정 후 중요 자산 보호를 위해 위험 수용·위험 회피·위험 전이·위험 감소 등의 대응 전략을 수립.

#### 단계별 보안 대책(보안 마스터플랜)
- 1단계(전사적 정보보안 환경 조성) : 정보보호 관리체계 구축을 위해 기술적 대책 구현 전 관리적 부문을 중심으로 보안 정책·조직·교육·정기 감사 등 보안 대책을 우선 수립.
- 2단계(전사적인 보안 시스템 구축) : 기본 보안 인프라 위에 방화벽 도입·물리적 보안 강화·PC 보안 등 기술적 대응책을 분야별로 구현해 조직의 보안 경영 체계를 완성.
- 3단계(보안 감사 및 유지 관리) : 추진된 과제를 지속적으로 유지·점검·관리하며 모니터링·정기 보안 감사·침해 사고 예방 대응 활동을 수행.

#### 개선 방안 실효성 검증
- ISMS-P : 한국인터넷진흥원의 정보보호 및 개인정보보호 관리체계 운용 기준으로, 개선 상황 점검 시 보안 감사 적용 기준으로 활용.
- 데이터 품질 5요건 : 완전성·유효성·정확성·유일성·일관성을 중심으로 데이터 품질을 점검.
- 기능 테스트 : 솔루션별 단위 테스트·PC 성능 영향도 점검·솔루션 간 충돌 검증 등 보안 기능을 검증하는 테스트.
- 시스템 통합 테스트 : 솔루션의 통합·연동성과 보안 운영 정책 정합성을 검증하여 기존 시스템과의 상호 운용성을 확인하는 테스트.
- 파일럿 테스트 : 현업 TFT 요원이 수행하여 운영 환경을 사전 검증하는 테스트.
- SLA 지표 : 개선 방안 도입에 대한 성과를 측정하는 데 활용하는 운영 성과 지표.

### 2-2. 시스템 보안 솔루션 운영 개선

#### 취약점 평가
- 개념 : 위협과 마찬가지로 취약점도 평가하며, 정보 자산에 악영향을 끼치는 위험 평가에 활용.
- 취약점 평가 기준(4단계) : 매우 취약(VH, 복구 불가/피해 큼), 비교적 취약(H, 복구 가능하나 피해 큼), 보통(M, 보고 필요), 취약하지 않음(L, 보고 불필요).
- 발생 가능도 : 취약성과 위협을 결합해 모호함·중복성을 제거한 취약점 평가 등급. 정보 자산 가치를 고려하며 높음·중간·낮음 세 등급으로 구분.

#### 보안 위험 분석 방법론
- 베이스라인 접근법 : 모든 시스템에 보호 기본 수준을 정하고 보호 대책을 선택. 장점-시간·비용 절약, 단점-조직 특성 미반영·수준 초과/미달.
- 비정형 접근법 : 전문가의 지식·경험으로 위험 분석. 장점-시간·비용 절약·소규모 적합, 단점-구조화 불가·합리적 도출 불가·사후 관리 제한.
- 상세 위험 분석 접근법 : 자산 가치·위험·취약성을 분석해 위험 정도를 결정. 장점-적절한 보안 수준 마련, 단점-전문 지식 필요·시간·비용 소모 큼.
- 복합 접근법 : 상세 위험 분석+베이스라인 복합 적용. 장점-빠른 전략 구축·효율적, 단점-적용 대상 설정 어려움·자원 낭비. 위험도 높은 대상은 상세 분석, 낮은 대상은 베이스라인 적용.

#### 운영 환경 모니터링
- SIEM : 중요 정보 자산의 보안 위험을 주기적으로 모니터링하기 위해 도입할 수 있는 통합 모니터링 솔루션.

## 학습 3. 시스템 보안 신규 위협 대응하기

### 3-1. 시스템 보안 위협 식별 및 차단

#### 보안 솔루션 관리 및 보호
- 보안 취약성 진단의 개념 : 정보 시스템의 보안상 문제점을 파악하고 내·외부 침해로 발생 가능한 위험 수준을 진단하는 작업.
- 보안 취약성 진단의 목적 : 안티바이러스(사후 조치)와 달리 사전에 침해사고를 예방하는 적극적 보안 행위. 가용성·무결성을 보장하는 보안 운영 기법.
- 보안 취약성 진단 및 조치 : 운영 환경에 맞게 진단 주기를 적정 수준으로 조정해 지속 관리.
- 위협 인지 및 대응 : 단순 공격 탐지가 아니라 취약성 파악 후 공격을 탐지하고 임팩트 분석(impact analysis) 수행.
- 최근 공격 경향 분석 : 최신 취약성·공격 여부를 분석하고 자사 존재 여부·탐지·대응 방법 파악.

#### 보안 위협 차단을 위한 시스템 보안 강화
- 불필요한 서비스 제거 : 원격에서 루트 권한을 취득할 수 있는 popd·imapd·rpc.cmsd 등의 서비스나 rsh·rlogin·rexec 등 'r' 명령어를 필요하지 않으면 제거하고, /etc/inetd.conf 등에서 불필요한 데몬을 주석 처리하는 보안 조치.
- setuid·setgid 설정 감사 : setuid·setgid를 이용한 루트 권한 취득 해킹이 가능하므로 해당 권한이 설정된 파일을 찾아 비인가된 사용 여부를 확인하는 파일 시스템 보안 조치.
- 원격지 루트 접속 제한 : 리눅스는 /etc/securetty에서 로컬 터미널만 남기고, 유닉스는 /etc/default/login의 CONSOLE 변수로 콘솔에서만 root 접속이 가능하도록 제한.

### 3-2. 복구 및 문서화

#### 정보보호 침해 사고
- 법적 정의 : 정보통신망법 제2조에 따라 해킹·컴퓨터 바이러스·논리 폭탄·메일 폭탄·서비스 거부·고출력 전자기파 등으로 정보 통신망·정보 시스템을 공격해 발생한 사태.
- 실무 정의 : 해킹·바이러스에 한정하지 않고 모든 전자적 공격 행위와 그 피해를 포함.
- 침해 행위에 따른 유형 : 비인가 시스템·파일 접근(바이러스·트로이잔·웜·백도어·악성코드), 비인가 네트워크 정보 접근, 비인가 서비스 이용, 서비스 방해·거부.
- 피해 유형 : 기밀성 침해(고객·기밀 정보 유출), 가용성 침해(서비스 지연·중단), 무결성 침해(정보 변조).

#### 침해 사고 분석 절차 (7단계, KISA 침해사고분석절차서)
- 사고 대응 전 준비 : 사고 전 대응팀·조직적 전략·대처 방안 개발.
- 사고 탐지 : 정보보호·네트워크 장비로 이상 징후 탐지(미생성 계정 발견, 디폴트 계정 로그인, 비정상 시간 활동, 권한 상승, 로그 삭제, 성능 저하 등).
- 초기 대응 : 대응팀 소집, 사고 정황 수집·기록(사건 유형 식별·영향 평가).
- 대응 전략 수립 : 환경 고려 사항을 반영해 적절한 대응 전략 수립.
- 사고 조사 : 호스트 기반·네트워크 기반 증거로 나누어 데이터 수집(디지털 포렌식)·분석(증거 훼손 방지).
- 보고서 작성 : 육하원칙에 따라 원인·분석 보고서 작성.
- 복구 및 해결 과정 : 위험 우선순위에 따라 복구 절차 마련·복구·사후 조치.

#### 침해 사고 탐지·조사·복구 보조 개념
- 사고 탐지 영역 : 침해 사고를 탐지하는 주체로 IDS·최종 사용자·네트워크 관리자·시스템 관리자·보안·인사부가 해당.
- 트립와이어(tripwire) : 윈도우즈 등 파일의 무결성을 점검하는 무결성 점검 도구.
- SAM(security account manager) : 윈도우즈 침해 사고 원인 확인 시 점검하는 계정 정보 데이터베이스.
- DR 시스템 : 침해 사고 및 IT 재해 복구를 위해 복제 백업한 데이터·애플리케이션 등을 이용하여 시스템을 복구하는 솔루션.
- 정적 분석 결과 : 사고 최초 탐지 시의 상태·로그 기록과 초기 분석 결과, 탐지된 공격 로그 등을 기록한 침해 사고 보고서 항목.
- 동적 분석 결과 : 공격자 활동 상세 분석·공격 프로그램 분석·네트워크 모니터링 결과 등을 기술한 침해 사고 보고서 항목.
- 재발 방지 대책 요청 : 발견된 보안 솔루션 취약점을 솔루션 벤더에 보고하여 패치 등 재발 방지 대책을 요청하는 사후 조치.

## 학습 4. 시스템 보안 솔루션 업데이트 적용하기

### 4-1. 시스템 보안 솔루션 업데이트 계획 수립

#### 패치 관리 시스템(PMS)
- 개념 : 서버 등 정보 자산에 설치되지 않은 윈도우 보안 패치 등을 검색해 자동 설치하는 프로그램/솔루션.
- 구성 : 패치 프라임 서버(패치 파일·배포 정보 보관·전달, 로그 수집, 웹 관리 페이지), 패치 배포 서버(배포 대상에 패치 전달, 로그 수집).
- 패치 관리 시스템의 보안 위협 유형 :
  - 패치 업데이트 서버 장악 : 해킹으로 침투해 설정 정보·파일 위·변조(예: 7.7 DDoS 2009).
  - 패치 서버 악성 코드 업로드 : 서버 장악 없이 악성코드를 업로드해 단말로 유포(예: 3.20 전산망 대란 2013).
  - 중간자(MITM) 공격 : 서버-단말 통신 경로에서 설정 정보·파일 위·변조.
  - 위장 서버 접속 : DNS 응답 위·변조로 위장 서버 접속을 유도해 악성코드 전송(예: SK컴즈 2011).
- 대응 방안 : 패치 서버 보안(파일 변조 방지·보안 강화), 채널 보안(고윳값/PKI 상호 인증).

#### 패치 관리 시스템의 보안 강화 대책 (파일·채널 보호)
- 파일 체크섬 : 패치 파일의 위·변조를 막기 위한 파일 보호 대책의 하나.
- 파일 암호화 : 패치 파일을 암호화하여 보호하는 파일 보호 대책.
- 디지털 서명 : 패치 파일의 무결성을 보장하기 위한 최고 수준(H)의 파일 보호 대책.
- 고윳값 상호 인증 : 패치 채널을 보호하기 위한 채널 상호 인증 방식.
- PKI 상호 인증 : 공개 키 기반의 채널 상호 인증으로 채널 보안 취약점 공격을 방어하는 대책.
- 대규모 MITM : DNS 포이즈닝·라우터 장악·공개 프록시 공격을 이용한 대규모 중간자 공격.
- 소규모 MITM : 스위치 장악·ARP 스푸핑을 이용한 소규모 중간자 공격.

### 4-2. 시스템 보안 솔루션 업데이트 적용

#### 업데이트와 업그레이드의 비교
- 업데이트(패치) : 소프트웨어에 변화를 주는 것. 최신 버전으로 끌어올리거나 오류를 고쳐 기능을 향상. SW에만 사용.
- 업그레이드 : 새로운 버전 또는 이전보다 나아진 완전히 새로운 것으로 교체. SW·HW 모두 사용(주로 하드웨어 교체).

#### 시스템 보안 솔루션의 보안 업데이트
- 발생 원인 : 솔루션 설계상 결함·개발상 결함으로 취약점이 발생하며, 무결한 코드는 존재하기 어려워 주기적 확인·업데이트 필요(CVE 웹사이트에서 발견된 취약점 목록 확인).
- 보안 업데이트 프로세스 CSF : 영향받는 자산 파악, 서비스 중요도에 따른 단계적 업데이트, 업무 의존 시간 내 적용, 테스트 서버 선적용.
- 솔루션 업데이트 체계 동작 구조 : ① 설정 정보 다운로드(최신 버전·URL 목록) → ② 업데이트 파일 다운로드 → ③ 대상 파일 설치.

#### 업데이트 전 백업 형태
- 특정 파일 및 폴더 백업 : 특정 파일 또는 폴더를 별도 저장 장치에 백업하는 형태.
- 전체 시스템 초기 백업 : 솔루션이 설치된 장비의 출하 시 이미지 전체를 백업하는 형태.
- 시스템 변경 사항 백업 : 시스템에 변경한 사항을 백업하여 스냅 샷 형태로 시스템 복구 지점을 생성하는 형태.
