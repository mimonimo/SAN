# 보안로그 분석 — 개념 정리

> 학습모듈: 보안시스템 로그를 분석하고, 보안로그 상관분석을 수행하며, 탐지패턴을 개발·최적화한다.

---

## 학습 1. 보안시스템 로그 분석하기

### 1-1. 보안로그 확인

#### 보안로그
- 로그(Log)의 개념 : 특정 사건(Event) 발생 시 발생 시점·내용·발생 주체 등의 정보를 기록한 것. 정책에 해당/위반되는 사건이나 장애 등 목표한 사건의 상세 내역을 기록.
- 보안로그(Security Log)의 정의 : 로그 중 "보안사건에 해당하는 로그". 보안정책에 해당하는 사건에 대한 기록으로, 주체·객체·발생 시간·행위·영향 범위 등의 정보를 포함.

#### 사이버 공격
- 개념 : 인터넷을 통해 이루어지는 외부의 인가되지 않은 공격으로, 내부 중요 정보가 변조·삭제·유출되는 현상.
- 사이버 공격 유형 : 홈페이지 악성코드 유포, 홈페이지 변조, DDoS, 피싱(Phishing), 파밍(Pharming), 스미싱(Smishing).
- 사이버공격 발생 순서 : 자산(가치+강점/취약점 동시 존재) 인지 → 자산 특성 조사·정보 수집 → 취약점 확인 → 악용 도구(Tool) 확보 → 취약점 공격.

#### 침해대응 절차
- 개념 : 침해(보안)사고 발생 시 대응하는 일련의 과정과 수행 행동. 사고 유형별 세부 대응 지침 마련 필요.
- 수립 목적 : 이해관계자별 역할·책임 명시로 신속한 업무 복구 유도, 조치 업무 순서·책임 범위 규정.
- 침해사고 대응 절차 : 사고 전 준비 → 사고 탐지(이상 징후·정탐/오탐 확인) → 초기 대응(세부사항 기록·관련 부서 통지) → 대응 전략 수립(전파·후속 공격 차단) → 사고 조사(데이터 수집·분석·복구) → 종료(보고서 작성·개선안 도출).

### 1-2. 보안로그 분석

#### 보안사건 판단
- 개념 : 침해사고 발생 시 해당 접근이 악의적 공격인지 정상 접근인지를 분석하는 행위.
- 보안사건 탐지 : 정책(규칙)과 일치 여부 확인 → 일치 시 '탐지', 불일치 시 '미탐지'. 규칙에 없으나 악의적 공격이면 Zero-day 공격으로 분류해 담당자가 직접 차단.
- 보안사건 탐지 분류 :
  - 정탐 : 정해진 규칙에 따라 보안사고를 정상 탐지(침해 사건을 보안사고로 판정, 비침해를 비사고로 판정).
  - 오탐 : 탐지 오류(보안사고가 아닌데 사고로 판정, 침해사건을 비사고로 판정).
- 오탐 유형 :
  - FAR(False Acceptance Ratio) : 악의적 공격을 정상 접근으로 판단하는 오탐. 규칙 부재·과도한 예외처리 시 발생, 보안 요구사항 높은 조직에서 발생.
  - FRR(False Reject Ratio) : 정상 접근을 보안 사건으로 판단하는 오탐. 규칙 오설정·예외처리 미적용 시 발생, 보안 요구사항 낮은 조직에서 발생.
  - ERR(Equal Error Rate, =CER) : FRR과 FAR이 동일하여 탐지 효율이 가장 높은 상태.

#### IP Packet
- IP(Internet Protocol) 개념 : 인터넷상의 한 컴퓨터에서 다른 컴퓨터로 데이터를 보내는 프로토콜. OSI 7계층 중 Layer 3의 대표 프로토콜.
- IP Packet 구조 : L2 Header(LAN용, Ethernet/IPX) + L3 Header(WAN용, 논리적 식별자·라우팅, IPv4/IPv6) + L4 Header(네트워크 제어·세션, TCP/UDP/ICMP) + Data(실제 사용자 데이터).
- 구조 이해의 의미 : IP Packet은 가장 많이 사용되는 프로토콜이므로 악의적 공격도 이를 사용. 각 Layer의 공격 기법·원리 이해가 필요.

## 학습 2. 보안로그 상관분석하기

### 2-1. 시스템별 보안로그 분석

#### 보안시스템 선별
- 보안로그 기록 : 대부분의 정보시스템은 보안사고 여부와 무관하게 로그를 기록하며, 그중 보안사건에 해당하면 보안로그로 간주.
- 보안시스템 유형 :
  - 네트워크 : 방화벽(정책 기반 트래픽 허용/차단·URL 차단·정보유출 차단), VPN(암호키 관리·암복호화·Replay 감지), IPS/IDS(악성코드 탐지·차단·격리), NAC(인증·비인가 시스템 차단).
  - Platform : Vaccine(악성코드 탐지·차단·복구), PMS(OS 패치·비인가 SW 탐지), 키보드 보안(키 입력 암호화), SecureOS(자원 비인가 접근 차단·비정상 프로세스 탐지).
  - Application : DRM·DLP(전자문서 생성/수정/출력 통제), USB 보안(분실·장착 식별·데이터 삭제), PKI(인증서 발행/폐기/갱신).
- 보안로그 기록 방식 :
  - 저장 Type : File(저장 용량 최소화·저용량 시 검색 속도↑), Database(저장 공간 증대·대용량 시 검색 속도↑·암호화·통합 저장).
  - 저장 위치 : Local Storage(내부 File System 활용, 장치 고장 시 활용 불가), 통합 로그 서버(네트워크로 다수 시스템 로그를 한 서버에 저장·연관 분석·수평 확장 가능).
- Platform별 보안로그 : 운영체제·로그 종류에 따라 저장 방식(File/Text, File/Binary)·조회 방법(제어판, GUI, Shell)이 상이.

### 2-2. 상관분석

#### 공격경로 확인 기법
- 공격경로의 개념 : 외부 악의적 행위자가 내부 시스템에 실행한 공격의 경로를 확인·도식화하는 행위. 공격 기법·도구·지원 시스템·플랫폼도 함께 조사.
- 기본 공격경로 확인 기법 : 공격 주체 확인, 단순 Routing 경로·Tracerouting 확인(1차 공격지까지 경로). 공격 주체 검증은 어려움.
- 상세 공격경로 확인 기법 : 공격 주체 시스템에 직접 접속해 로그·행위 분석. 실제 공격 주체 여부·경로 시스템 여부 검증, 공격 기법·취약점 확인 가능.
- 확인 시 주의사항 : 제2~3국 공격 주체 확인은 수사 협조 필요(미협조 시 국제법 위반), 주체 특정 검증은 Terminal Service·Relay Tool·계정 도용 여부까지 반복 검증.

#### 피해분석 범위
- 피해분석의 개념 : 침해사고로 인한 내부 전체 자산의 피해 정도를 확인하는 행위. 물리적 피해뿐 아니라 논리적 피해·복구 비용을 포괄.
- 피해분석 범위 유형 : 직접적 피해(물리적 피해-시스템 손상·교체, 논리적 피해-연계·지원 정도 대상), 간접적 피해(보안솔루션 도입 비용, 소송 대응·보상 등).

## 학습 3. 탐지패턴 개발하기

### 3-1. 탐지패턴 개발

#### 탐지패턴 개발
- 개발 목적 : 새롭게 발견된 공격이 재발할 경우 즉각 탐지·차단하기 위해 탐지패턴을 개발·적용.
- 탐지 모형 :
  - 이상 탐지 : 정보·정보 유입 순서가 규칙에 부합하는지 확인, 규칙 위반 시 보안사건으로 간주(예: 규칙 i=정수인데 i="A"이면 이상 행위).
  - 오용 탐지 : 정보 활용 정도를 확인, 접근/반복 시도가 임계치를 초과하면 위반으로 간주(예: 규칙 1Gbps인데 5Gbps 유입이면 오용 행위).
- 탐지패턴 개발 프로세스 :
  - 공격 기법 분석 : 공격의 유니크한 특징 도출(공격 순서·메카니즘 분석, 영향받는 시스템·SW 식별, 피해 정도 산정).
  - 탐지패턴 개발 : 적용 보안시스템 선정, 적용 가능한 규칙(정규식 등) 선정, 공격 특징 표현, 보안시스템에 적용.
  - 탐지패턴 검증 : 동일·유사 공격으로 정탐률·오탐률 검사, 반복 수행으로 최적화.
  - 탐지패턴 배포 : 동일·유사 보안시스템에 전달·적용.
- 탐지패턴 개발 도구 : 보안로그(출발지/경유지 IP·국가·Port·프로토콜·Packet 내용 도출), 공격도구(공격 Sequence·메카니즘 확보).

---

### 3-2. 탐지패턴 최적화

#### 탐지패턴 최적화
- 최적화 목적 : 오탐률 최소화·정탐률 증가, 탐지 Resource 소모 최소화·속도 증가, 업무 지속성 보장·외부 공격 차단의 개선 과정.
- 탐지패턴 최적화 프로세스 :
  - 탐지패턴 적용 : 개발된 패턴을 보안시스템에 업로드·적용(Monitoring만 적용 가능).
  - 탐지율 평가 : 인위적으로 동일 침해사고를 발생시켜 탐지 여부 확인, 정상·유사 접속으로 오탐 확인(최소 1일 이상 지속 평가).
  - 원인 분석 : 패턴 분석·업무 시스템 환경 분석·오탐 Traffic과 현행 패턴의 Gap 분석.
  - 탐지패턴 개선 : 탐지 범위·표현식 변경 및 재적용.
- 최적화 주의사항 : 초기 패턴은 Monitoring만 적용(업무 지장 방지), 탐지 범위 과대 설정 시 오탐↑, 과소 설정 시 미탐↑, 표현식 오류 시 Resource 과소모·성능 저하.

#### 탐지 성능 개선
- 배경 : 개발 직후 탐지패턴은 오탐률·Resource 소모량이 높아 효율이 떨어짐.
- 목적 : 탐지 범위·방법을 변경하여 탐지 효율·성능을 높임.
- 탐지 성능 개선 기법 :
  - 탐지 범위 조정 : 범위 축소(미탐률↓·오탐률↑), 범위 확대(정탐률↑·Resource 사용↑).
  - 탐지 표현식 변경 : Converting 과정을 최소화하여 탐지 효율을 높임.
  - 탐지 순서 변경 : 조합 패턴을 동시 매칭/단일 패턴으로 변경해 성능 향상.
  - 탐지 적용 대상 변경 : 가변 범위를 모두 포함한 패턴으로 변경해 Skip 과정에 의한 성능 저하 방지.
