# 애플리케이션 보안 운영 — 개념 정리

> 학습모듈: 애플리케이션 보안 솔루션을 운영·개선하고, 신규 위협에 대응하며, 솔루션 업데이트를 적용한다.

---

## 학습 1. 애플리케이션 보안 솔루션 운영하기

### 1-1. 애플리케이션 보안 솔루션 요구사항 분석

#### 정보보호 관리체계
- 개념 : 정보시스템·정보 통신망의 안정성·신뢰성 확보를 위해 관리적·기술적·물리적 보호를 포함하는 종합적 관리체계.
- ISO 27001 : ISO 27000 시리즈 중 하나로 ISO/IEC JTC1 SC27 WG1에서 추진. 정보보호 요구사항을 4개 도메인, 93개 통제 항목으로 구성.
- ISMS-P : 주요 정보 자산을 보호하기 위한 정보보호 관리체계가 인증 기준에 적합한지 심사·인증하는 제도. 102개 인증 기준으로 심사.
  - 관리체계 수립 및 운영 : 관리체계 운영 라이프사이클(메인 프레임). 4개 분야 16개 인증 항목.
  - 보호대책 요구사항 : 정책·조직·교육 등 관리적 부문과 개발·보안 통제·운영 통제 등 물리적·기술적 부문. 12개 분야 64개 인증 항목.
  - 개인정보 처리 단계별 요구사항 : 수집·이용 및 제공·저장 및 보관·파기의 개인정보 생명주기. 5개 분야 22개 인증 항목.

#### 정보보호 관련 법률과 기관의 규정·지침
- 정보보호 관련 법률 : 정보통신망법, 정보통신기반보호법, 전자정부법, 전자서명법, 정보보호산업진흥법, 개인정보보호법. 산업 분야별로 금융(전자금융거래법·신용정보법), 의료(의료법·보건의료기본법), 제조(산업기술유출방지법) 등이 있음.
- 기관의 규정 및 지침 : 각 기관별로 정보시스템 운영·보안 관련 규정, 개인정보 처리 규정 등 내부 운영 규정·지침.

#### 애플리케이션 현황 구분
- 망별 구분 : 내부망·외부망 등 해당 기관의 정보 통신망 구성에 따라 애플리케이션 구조를 구분.
- 용도별 구분 : 대외 서비스용·내부 업무용·정보시스템 관리용 등으로 구분(중요 서비스용 여부에 따라 정보보호 우선순위를 차등).
- 운용 현황별 구분 : 상시 운용·비정기 운용·운용 중지 등으로 구분(운용 현황에 따라 모니터링 주기·방법이 달라짐).
- 자산 가치(중요도) : 자산을 기밀성(C)·무결성(I)·가용성(A) 등으로 평가하여 상·중·하 등급으로 산정한 자산의 중요도.

#### 애플리케이션 보안 요구사항 구분
- 기능 요구사항 : 기밀성·무결성을 유지하기 위한 보안 관련 기능적 요소의 요구사항(인증 대상·방식, 인증 수행 제한, 암호 키 관리, 중요 정보 전송 등).
- 화면 요구사항 : 보안 요구사항을 구현한 화면 및 UI에 관한 요구사항(예외 처리 오류 메시지 노출, 비밀번호 표시 제한 로그인 UI 등).
- 성능 요구사항 : 보안 기능 또는 보안 솔루션의 성능에 관련된 요구사항(파일 암호화 처리 속도, 로그인 응답 속도 등).
- 기타 요구사항 : 그 밖의 보안 요구사항(유지 보수·온라인 기술 지원, GS·SP 인증 등 SW 관련 인증 요구사항 등).
- 보안 요구사항 유형 : 정보를 애초에 취득하지 못하게 하는 처리 제한, 이미 취득한 정보의 접근을 제한하는 접근 통제, 접근하더라도 기밀성을 유지하는 암호화 세 가지로 구분.

### 1-2. 애플리케이션 보안 솔루션 운영

#### 애플리케이션 보안
- 개념 : 애플리케이션은 상용 SW 솔루션, 웹 애플리케이션을 포함하여 SI 구축 방식으로 개발된 모든 응용프로그램.
- 애플리케이션 공격 유형 (대표 웹 공격, OWASP Top10 참고)
  - XSS(Cross Site Scripting) : XSS 취약점을 이용해 악성 스크립트를 서버측·URL에 삽입하고, 피공격자의 브라우저에서 실행되게 하는 공격.
  - SQL Injection : 입력값·URL 파라미터에 악의적 시스템 명령·SQL 구문을 삽입(로그인·DB 열람·명령 권한 획득).
  - CSRF(Cross Site Request Forgery) : 사용자가 인지하지 못한 상황에서 사용자 의도와 무관하게 공격자가 의도한 행위(수정·삭제·등록)를 요청하게 하는 공격(특히 관리자 권한 악용 시 치명적).
- 애플리케이션 보안 프로토콜 (OSI 7계층 중 응용 계층)
  - SSH(Secure Shell) : 원격 단말 접속·명령 실행·파일 전송을 안전하게 해 주는 프로토콜. 텔넷·rsh·rlogin을 대체, 암호화로 기밀성 유지.
  - S/MIME : 전자 우편 내용 보호를 위한 수단으로 SMTP를 대체. 디지털 서명과 메시지 암호화 기능 제공.

#### 웹 애플리케이션 취약점 점검 항목
- 디렉터리 인덱싱 노출 취약점 : 웹 애플리케이션 경로를 직접 입력해 접근 시 웹 서버 디렉터리·파일 목록이 출력되면 존재하는 취약점.
- 관리자 페이지 노출 취약점 : admin·manager·webmaster 등 추측 가능한 관리자 페이지 경로로 직접 접근 시 관리자 페이지에 접속되면 존재하는 취약점.
- 위치 공개 취약점 : 추측 가능한 백업 파일명이나 디렉터리 인덱싱을 통해 백업 파일이 노출되면 존재하는 취약점.
- 웹 서비스 메소드 설정 공격 : delete·put 등 악의적으로 사용할 수 있는 메소드 요청이 허용되면 존재하는 취약점.

## 학습 2. 애플리케이션 보안 솔루션 운영 개선하기

### 2-1. 애플리케이션 보안 솔루션 운영 개선 방안 수립

#### 애플리케이션 보안 솔루션 (종류)
- 방화벽(Firewall) : 서로 다른 네트워크를 지나는 패킷을 허용/거부하는 HW 또는 SW.
- 침입 차단 시스템(IPS) : 외부→내부 유입 패킷 중 악의적 패킷을 찾아 제어하는 HW/SW.
- 가상 사설망(VPN) : 전용 회선을 조직 내부 사용자를 드러내지 않고 공유하기 위한 사설 통신망.
- 네트워크 접근 통제(NAC) : 내부망 호스트가 네트워크 접근 시 보안 정책에 따라 접속을 제어·통제하는 HW/SW.
- 시스템 접근 제어 : 시스템(서버)에 접근 권한·경로·실시간 감시를 위한 HW/SW.
- DB 암호화 : DB 저장 데이터의 기밀성 유지를 위해 암·복호화를 수행하는 HW/SW.
- 자료 유출 방지(DLP) : 내부 정보 유출을 차단하고 개인정보를 보호하는 HW/SW.
- 안티바이러스 : 일반적으로 백신 프로그램으로 알려진 SW.

#### 보안 취약점
- 개념 : HW·SW의 결함이나 설계상 허점으로 권한 이상의 명령·정보 열람을 허용할 수 있는 약점.
- CVE(Common Vulnerabilities and Exposures) : 알려진 보안 취약점 정보를 공유·대처하기 위해 미국 MITRE 재단이 제공하는 취약점 목록 체계.
- 시큐어 코딩 : 개발 과정에서 휴먼 오류·논리적 오류로 발생 가능한 보안 취약점을 최소화하는 사전 예방 활동(국내 2012년 SW개발 보안 제도 도입, 공공부터 의무화).

#### 보안 취약점 진단·개선
- 포트 스캔 : 허용하지 않는 포트가 열려 있지 않은지 점검하는 방법.
- 취약점 심각도 구분 : 확인된 보안 취약점을 심각·보통·단순 등으로 구분하여 개선 우선순위를 정하는 것.
- 보안 취약점 개선 방안 구분 : 보안 설정 변경에 의한 개선과 보안 패치(업데이트) 적용에 의한 개선 등으로 구분(개선 내용 반영 여부를 확인할 수 있는 테스트 방안을 함께 작성).

### 2-2. 애플리케이션 보안 솔루션 운영 개선

#### 모의 해킹
- 개념 : 보안 취약점 점검을 위해 보안 담당자의 승인 아래 내·외부에서 침투 테스트를 시도하고 대응 방안을 마련하는, 악의적이지 않은 사이버 공격.
- 외부 모의 해킹 : 웹 애플리케이션의 인증·세션·암호화 전송·웹 보안·접근 제어 등을 외부에서 해커를 가장해 침투 테스트.
- 내부 모의 해킹 : 악의적 내부 사용자를 가장하여 내부망에서 정보 유출·해킹을 시도(내부망·내부 사용자 권한 사용 점에서 외부와 차이).

#### 애플리케이션 운영 서버 보안
- 개념 : 애플리케이션은 서버에서 작동되므로 운영 서버 보안이 가장 기본. 물리적 보안과 함께 보안 설정·불필요 서비스 중지·포트 점검·모니터링 등을 수행.
- 보안 서버 운영 : 인터넷 통신 시 개인정보 등 중요 정보를 암호화하여 송수신하도록 구축한 웹 서버. 운영 중인 웹 서비스에 SSL 인증서·암호화 SW를 설치해 송수신 암호화 제공.
- 시큐어 코딩 : 서버 동작 애플리케이션 SW 개발 시 발생 가능한 보안 취약점을 제거해 해킹을 예방. 주기적 점검으로 소스 코드 취약점을 수시 개선.

## 학습 3. 애플리케이션 보안 신규 위협 대응하기

### 3-1. 애플리케이션 보안 위협 식별 및 차단

#### 애플리케이션 보안 취약점과 위협 식별
- 구분 : 애플리케이션 보안 위협은 의도적 공격과 애플리케이션 자체에 내재된 보안 취약점으로 구분.
- 애플리케이션 보안 취약점 : 개발 단계의 개발자 실수·구조적 결함 등에 의한 취약점.
- 애플리케이션 관련 보안 위협(의도적 공격) :
  - 해킹 : 악성 프로그램·백도어 등 공격 기법이나 취약점을 이용해 정보시스템을 공격.
  - 피싱 : 전자우편·메신저 등으로 신뢰 대상으로 위장하는 사회 공학적 공격.
  - 스팸 : 불특정 다수에게 광고성 정보를 송신, 특정 사이트 접속을 유도해 악성코드 감염에도 이용.
  - DoS/DDoS : 좀비 PC로 정보시스템 자원을 고갈시켜 서비스 불가 상태로 만드는 공격.
- 애플리케이션 보안 취약점 7대 유형 : 입력 데이터 검증 및 표현(SQL 인젝션·XSS), 보안 기능(부적절한 인가·취약한 암호화), 시간 및 상태(제어문 없는 재귀함수), 에러 처리(오류 메시지 통한 정보 노출), 코드 오류(타입 변환·자원 반환 오류, 널 포인터 역참조), 캡슐화(잘못된 세션에 의한 정보 노출), API 오용(DNS lookup 의존 보안 결정).

#### 애플리케이션 보안 취약점 점검 도구
- Nikto : Perl 기반의 널리 알려진 무료 웹 취약점 스캐너.
- Paros : JAVA 기반으로 HTTP·HTTPS를 지원하며 Proxy·XSS·SQL Injection 점검과 HTML 리포트를 제공하는 Proxy 도구.
- Wikto : Nikto의 윈도우 버전으로 Google API를 제공하는 점검 도구.
- Absinthe : Blind SQL Injection을 점검하는 SQL Injection 스캐너.
- AD 주입 도구 : 링크를 따라가며 취약한 페이지를 확인하는, 비교적 정확한 결과를 얻는 SQL Injection 스캐너.
- NBSI : SQL Injection 공격 도구.
- nmap : 포트가 열려 있는지 점검할 수 있는 오픈소스 포트 점검 도구.

#### 네트워크 기반 위협 식별
- 망 분리 : 외부망과 내부망을 구분하여 구성함으로써 악의적 공격으로부터 보호 영역과 노출 영역을 식별하는 것.

### 3-2. 복구 및 문서화

#### 애플리케이션 보안 취약점 확인 (취약점 정보 체계)
- CVE(Common Vulnerabilities and Exposures) : 알려진 보안 취약점 목록 체계. 'CVE-연도-등록번호' 형식(2015년부터 자릿수 제한 없음). MITRE 제공.
- CWE(Common Weakness Enumeration) : 일반적 취약점을 열거해 SW 취약점 유형을 정형화한 목록을 개발자·보안 관계자에게 제공. MITRE 제공.
- CWE/SANS Top25 : CWE 중 치명적 약점을 정리한 목록. 프로그래밍·설계·아키텍처 오류 자료를 제공해 사전 예방.
- OWASP Top 10 : MITRE·PCI DSS·DISA·FTC 등이 참여해 애플리케이션 보안 위험 상위 10개를 선정·발표.

## 학습 4. 애플리케이션 보안 솔루션 업데이트 적용하기

### 4-1. 애플리케이션 보안 솔루션 업데이트 계획 수립

#### 애플리케이션 보안 솔루션 업데이트 분류
- 보안 기능 향상을 위한 업데이트 : 웹 방화벽·유해 사이트 차단 시스템 등은 신규 공격 패턴·악성 URL 추가 시마다 주기적 업데이트 필요.
- 운영 SW 오류 및 보안 취약점 제거 : 솔루션 구현 SW의 오류·취약점이 발견되면 공격자가 악용·우회할 수 있으므로 최신 버전으로 업데이트.

#### 애플리케이션 보안 솔루션 업데이트 방식
- 제조업체 패치서버를 통한 온라인 업데이트 : 제조업체 전용 패치서버로 온라인 업데이트(방화벽 접근 통제 권장). 안전·신속하나 서버 해킹·무결성 미확인 시 모든 사용자 피해 우려.
- 제조사 홈페이지 등을 통한 오프라인 업데이트 : 홈페이지·메일로 패치파일을 별도 제공받아 업데이트. 다소 불편하나 무결성 검증 후 적용해 더 안전. 솔루션이 여러 개면 PMS로 업데이트 가능.

#### 패치 관리
- PMS(Patch Management System) : 안전하게 검증된 패치파일을 조직 내 여러 개의 보안 솔루션에 적용·업데이트할 수 있는 패치관리시스템.
- 패치 관리 절차 : 신규 업데이트 정보 수집 → 애플리케이션 보호 대상 목록 작성 → 보안 솔루션 목록·벤더 패치 방법 정리 → 최신 보안 취약성·패치 확인 → 시스템 환경 분석 → 패치 적용 계획 수립 → 패치 테스트 → 패치 적용 → 이상 현상 모니터링의 일련의 절차.

### 4-2. 애플리케이션 보안 솔루션 업데이트 적용

#### 애플리케이션 보안 솔루션 업데이트 체계
- 개념 : 솔루션 SW의 기능 개선·버그·취약점 수정을 위해 벤더사가 내놓는 프로그램. 업데이트 체계는 취약점 확인→수정 SW 검색·다운로드→설치의 과정. 구성: 업데이트 중앙 서버(벤더), 업데이트 관리 서버(기관), 클라이언트 장치.
- 업데이트 유형 :
  - 온라인/오프라인 : 수동 업데이트(오프라인, 직접 다운로드·설치), 자동 업데이트(온라인, 일정에 따라 검사·다운로드·설치). 최근 자동 업데이트가 일반적.
  - 중요도 : 중요 업데이트(취약점 패치·중대 문제 개선 등 필수), 선택적 업데이트(작업 환경 향상·드라이버·신규 SW 등 선택).
- 자동 업데이트 동작 구조 : ① 중앙 서버→관리 서버로 패치 정보(버전·URL 목록) 송신 → ② 최신이 아니면 파일 다운로드 → ③ 패치 전 설정값 등 중요 정보 백업 → ④ 업데이트 대상 파일 설치.

#### 업데이트 적용·복구
- 업데이트 사전 테스트 : 실제 운영 환경과 동일한 환경(개발 환경·테스트 시스템)에서 업데이트를 미리 적용해 오류 발생·정상 작동 여부를 확인하는 것.
- 롤백(rollback) : 업데이트가 정상적으로 적용되지 않을 경우 사전에 백업한 데이터를 이용해 이전 상태로 되돌리는 작업.
