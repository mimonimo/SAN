# 정보 시스템 진단 — 개념 정리

> 학습모듈: 정보 시스템 진단 기준을 마련하고, 진단 결과를 도출하며, 대응책을 마련한다.

## 학습 1. 정보 시스템 진단 기준 마련하기

### 1-1. 정보 시스템 진단 준비 및 취약점 정보 파악

#### 정보 시스템의 개요
- 개념 : 정보 시스템은 기업이 다양한 의사 결정을 내리는 데 필요한 정보를 수집하고 그것을 목적에 따라 가공하며, 그 결과를 의사 결정자에게 제공하는 구조이다.
- 정보 시스템의 관리 요소(10가지) :
  - 구성 및 변경 관리 : HW·SW의 구성 현황·이력·구성 파일·파라미터·구성도를 관리하고 변경 사항을 효율적으로 관리하는 업무.
  - 운영 상태 관리 : 구성 요소의 운영 상태를 관리해 이상 징후를 발견·기록·분류·통지하여 가용성을 향상.
  - 성능 관리 : 효율·응답 속도를 최적 상태로 유지하기 위해 낮은 성능 요소를 찾아 개선.
  - 장애 관리 : SW·HW·DB·네트워크의 고장·장애·서비스 불능을 관찰·진단·보고·제어·처리하는 과정.
  - 보안 관리 : 정보 자산의 정보 보호 활동·절차를 체계적으로 관리해 기밀성·무결성·가용성을 확보하고 불법 유출·파괴·변경으로부터 보호.
  - 백업 관리 : 장애·재해로 정보 소실·손상에 대비해 데이터를 별도 매체에 예비 저장하여 정보 시스템의 영속성을 보장.
  - 사용자 지원 관리 : 운영 지침서 수정·보완, 교육, 요구 처리 등으로 최종 사용자 만족도를 향상.
  - 전산실 관리 : 전산실 접근 통제와 건물·설비 관리로 전산 시스템 운용의 안전성·신뢰성을 보장.
  - 운영 아웃소싱 관리 : 효율적·체계적 운영을 위해 외부 기관으로부터 운영 서비스를 획득하는 절차.
  - 예산 관리 : 운영 예산 소요를 예측하고 예산을 계획·집행·통제하는 절차와 지침.

#### 위험의 정의
- 개념 : 위험(risk)은 원하지 않는 사건이 발생하여 손실 또는 부정적 영향을 미칠 가능성. 손실 사건 발생 가능성과 발생 손실 정도에 비례.
- 위험 평가 산식 : 위험 = 발생 가능성 × 손실의 정도.
- 위험 평가 함수 : 위험 = f(자산, 위협, 취약성).

#### 위험의 구성 요소
- 자산(Asset) : 조직이 보호해야 할 대상(정보·HW·SW, 인력 등 무형 자산 포함). 데이터·문서·SW·HW·물리적·인적 자산 등으로 구분. 위험 분석 시 보안 사고 발생 시의 손실(loss)을 파악.
- 위협(Threats) : 자산에 손실을 초래할 수 있는 원치 않는 사건의 잠재적 원인. 환경적 요인(자연재해·장비 고장)과 인간에 의한 위협(의도적/우연적)으로 분류. 핵심 속성은 발생 가능성(연간 발생 횟수). 위협의 발생 가능성과 성공 가능성은 다를 수 있음.
- 취약성(Vulnerability) : 자산의 잠재적 속성으로 위협의 이용 대상, 또는 정보 보호 대책의 미비. 자산과 위협 사이의 관계를 맺어 주는 특성.
  - 취약성 속성 : 모든 자산에 잠재, 그 자체로는 위협을 초래하지 않고 위협에 이용될 때만 위험 발생, 대응책이 늘수록 감소, 대응책 자체도 취약성을 가져 0이 될 수 없음.
  - 취약성 유형 분류 : 관리적 측면(정책·조직·인사), 물리적 측면(건물·시설물), 기술적 측면(네트워크·시스템·DB·PC·응용 프로그램).

#### 자산 평가 및 위협 분석
- 영향도 파악 : 자산 가치 산정 시 단기적 손실과 장기적 영향을 포함하여 자산의 중요도 또는 사고 발생 시 미칠 수 있는 영향의 규모를 파악하는 것.
- 자산 중요도 산정 : 자산의 기밀성·무결성·가용성 점수를 합친 점수로 중요도 등급을 평가하며, 정성적 기준(기밀성·무결성·가용성)과 정량적 기준(구매 비용·침해 사고 피해 금액)을 사용.
- 자산 중요도 등급 전환 기준 : 자산의 중요성이 사용 용도·규모·업무 중요도에 따라 변경되므로 자산 중요도 등급의 변경 기준을 미리 설정해 두는 것.
- 위협 시나리오 : 파악되지 않은 위협을 발견하기 위해 위협 환경을 설정하고 발생 가능한 위협을 유추하여 작성하는 시나리오.
- 위협 발생 빈도 : 위협이 얼마나 자주 발생하는지를 나타내며, 통계 자료가 없는 경우 발생 가능성을 유추해 산정(피해 규모가 작아도 지속 발생 시 영향이 커질 수 있음).
- 피해 규모 : 위협이 자산에 미치는 손실의 크기로, 손실 정도와 업무 중단 여부에 따라 낮음·중간·높음·매우 높음으로 구분.

### 1-2. 정보 시스템 위험 평가 방법 및 보호 대책 파악

#### 위험 평가 방법론
- 위험 관리 : 자산의 위험을 수용 가능한 수준으로 유지하기 위해 위험을 분석하고 보호 대책을 마련하는 과정.
- 기준선 접근법(Baseline) : 모든 정보 자산에 보호 기본 수준을 정하고 보호 대책을 적용. 시간·비용 적게 들지만 부서별 보안 수준이 과대/과소할 수 있음.
- 비정형 접근법(Informal) : 전문가의 지식·경험으로 위험 분석(전문가 판단법). 작은 조직에 효과적이나 위험 평가·비용 도출이 어렵고 사후 관리 제한.
- 상세 위험 분석(Detail Risk Analysis) : 자산 가치·위협·취약성을 분석해 위험 정도를 결정. 적절한 보안 수준 마련 가능하나 전문 지식·시간·노력 소요.
  - 정성적 위험 평가 : 손실·위험을 개략적 크기로 비교, 상황 설명으로 묘사.
  - 정량적 위험 평가 : 위험을 손실액 등 숫자값으로 표현. 연간 기대 손실 = 위협 성공 시 예상 손실액 × 발생률.
- 복합 접근법(Combined) : 위험이 높은 시스템은 상세 위험 분석, 그 외는 기준선 접근법 적용. 신속한 보안 전략 수립·효율적.

#### 위험 처리 방법
- 위험 수용 : 현재의 위험을 받아들이고 잠재적 손실 비용을 감수(일정 수준 이하 위험 인정).
- 위험 감소 : 위험을 감소시키는 대책을 채택·구현. 비용 효과 분석으로 양(+)의 효과를 갖는 대책 선택(보호 대책 효과 = 원래 ALE − 대책 구현 후 ALE − 연간 대책 비용).
- 위험 회피 : 위험이 존재하는 사업을 수행하지 않고 포기.
- 위험 전가 : 보험·외주 등으로 잠재적 위험 비용을 제3자에게 이전.

#### 정보 보호 관리 체계
- 개념 : 조직 자산을 효율적으로 보호하도록 기술적·물리적·관리적·조직적 보호 대책을 구현하고 지속 관리·운영하는 종합적 체계. 기밀성·무결성·가용성·준거성 달성을 위해 위험을 평가하고 대책을 수립·운영.
- 관리 과정(5단계) : 정보 보호 정책 수립 → 정보 보호 관리 체계 범위 설정 → 위험 관리 → 구현 → 사후 관리.

#### 위험도 평가 및 위험 처리 전략
- 자산 그룹화 : 위험 분석을 위한 자산 분석 과정에서 보안 특성과 중요도가 동일한 같은 종류의 자산을 하나의 그룹으로 묶는 것.
- 침투 시험 : 기술적 취약성 점검 및 침투 시험을 수행하여 정보 시스템이 비인가된 논리적 접근 또는 사용자 신분 위장에 어느 정도 취약한지를 평가하는 것.
- 위험도 평가 : 자산 중요도와 발생 가능성에 따라 3단계(높음·중간·낮음) 또는 5단계(매우 높음~매우 낮음)로 평가하며, 위험도는 곧 대응해야 할 우선순위를 의미.
- 수용 가능 위험(DoA, Degree of Assurance) : 조직의 사업·업무 특성과 경영진 특성에 따라 결정되는 수용 가능한 목표 위험 수준으로, 기술적으로 결정될 수 없고 최고 경영진이 의사 결정해야 하는 사항.
- 목표 위험 수준 : 위험을 체계적으로 관리하기 위해 설정하는 기준 수준으로, 이보다 낮은 위험은 수용하고 그 이상의 위험은 목표 수준 달성을 위한 장기 계획을 수립해 대응.
- 위험 처리 전략 : 위험을 식별한 뒤 위험도가 목표 위험 수준 이하면 수용, 비용 효과적 대책이 존재하면 감소, 적절한 보험·전가 대상이 존재하면 전가, 그렇지 않으면 회피하는 절차.

## 학습 2. 정보 시스템 진단 결과 도출하기

### 2-1. 정보 시스템 취약점 진단 방법 파악

#### 서버 취약점
- 계정 관리 : root 계정 원격 접속 제한, root 계정 전환(su) 제한(무작위 대입·추측 공격 방지), root 외 동일 UID 금지.
- 파일 및 디렉터리 관리 : root 홈·PATH 권한 설정(PATH에 "." 포함 시 비의도적 명령 실행), 파일·디렉터리 소유자 설정, UMASK 설정 관리, 숨겨진 파일·디렉터리 검색·제거.
- 서비스 관리 : r 계열 서비스 비활성화, automountd 제거(RPC 악용 방지), DoS 취약 서비스 비활성화, RPC 서비스 확인(버퍼 오버플로 취약).

#### 네트워크 취약점
- 계정 관리 : 패스워드 복잡성 설정(무작위 대입 방지), 초기 설정 패스워드 변경.
- 접근 관리 : 불필요한 보조 입출력 포트 사용 금지, vty 접속 시 안전한 프로토콜 사용(Telnet은 평문 전송으로 스니핑에 노출).
- 패치 관리 : 최신 보안 패치 및 벤더 권고 적용.
- 로그 관리 : 정책에 따른 로깅 설정, 로깅 버퍼 크기 설정(초과 시 로그 손실).
- 기능 관리 : SNMP 서비스 확인(UDP 기반 공격 취약), Spoofing 방지 필터링 적용.

#### 애플리케이션 취약점
- 부적절한 에러 처리 : 다양한 에러를 유발해 에러 결괏값으로 웹 프로그램 구조·환경 설정을 추측해 공격.
- SQL 인젝션 : 입력값 유효성 검증 누락 시 조작된 SQL 쿼리로 인증 우회·정보 유출·시스템 명령 실행.
- 불충분한 인증 : 민감 데이터 경로의 인증 절차 부족으로 권한 외 페이지 접근·정보 유출·변조.
- 크로스사이트 스크립팅(XSS) : 게시판 등에 스크립트를 삽입해 열람자 브라우저에서 실행, 인증 정보 유출·악성코드 설치.
- 쿠키 변조 : 보호되지 않은 쿠키값 변조로 타 사용자 위장·권한 상승.
- 파라미터 변조 공격 : 클라이언트-서버 간 파라미터값을 조작해 중요 정보 조작·권한 획득.

#### 데이터베이스 취약점
- 계정 관리 : 기본 계정(system·scott 등) 패스워드·정책 변경, 불필요 계정(SCOTT·ADAMS 등) 제거·잠금, DBA 권한(SYSDBA)을 필요 계정에만 허용.
- 접근 관리 : 원격 접속 제한(미허용 IP 차단), DBA 외 시스템 테이블 접근 차단, 주요 설정·패스워드 파일 접근 권한 적절 설정.
- 옵션 관리 : 응용/DBA 계정 Role의 Public 설정 방지, OS_ROLES·REMOTE_OS_AUTHENT·REMOTE_OS_ROLES를 FALSE로 설정.
- 패치 관리 : 최신 보안 패치·벤더 권고 적용, 접근·변경·삭제 감사 기록을 정책에 적합하게 설정.

#### 서버 취약점 점검 항목(추가)
- NFS(Network File System) : TCP/IP를 사용해 원격지 컴퓨터의 파일을 자신의 컴퓨터에 있는 것처럼 검색·저장·수정하도록 하는 클라이언트-서버형 응용 프로그램으로, 환경 설정 오류부터 비인가 root 권한 획득까지 취약성 범위가 다양.
- Finger 서비스 : local 또는 원격 사용자의 계정 정보를 확인하는 명령으로, 호스트의 사용자 리스트 등 공격자에게 유용한 정보를 제공하므로 불필요하면 비활성화.
- Anonymous FTP : 익명 접속을 허용하면 악의적 사용자가 시스템 정보를 획득할 수 있고 디렉터리 쓰기 권한이 있을 경우 다양한 공격이 가능하므로 필요한 사용자만 접속하도록 설정.

#### 애플리케이션 취약점 유형
- 버퍼 오버플로(BO) : 메모리나 버퍼의 블록 크기보다 더 많은 데이터를 넣음으로써 결함을 발생시키는 취약점.
- 포맷스트링(FS) : 스트링을 처리하는 부분에서 메모리 공간에 접근할 수 있는 문제를 이용하는 취약점.
- LDAP 인젝션(LI) : LDAP 쿼리를 주입함으로써 개인 정보 등의 내용이 유출될 수 있는 문제를 이용하는 취약점.
- 운영 체제 명령 실행(OC) : 웹 사이트의 인터페이스를 통해 웹 서버를 운영하는 운영 체제 명령을 실행하는 취약점.
- SSI 인젝션(SS) : 서버가 HTML 문서에 입력하는 변수값(SSI)을 이용해 웹 서버상의 파일을 include시키고 명령문이 실행되게 하여 데이터에 접근하는 취약점.
- XPath 인젝션(XI) : 조작된 XPath 쿼리를 보냄으로써 비정상적인 데이터를 쿼리해 올 수 있는 취약점.
- 디렉터리 인덱싱(DI) : 요청 파일이 존재하지 않을 때 자동으로 디렉터리 리스트를 출력하는 취약점.
- 정보 누출(IL) : 개발 과정의 코멘트나 오류 메시지 등에서 중요한 정보가 노출되어 공격자에게 2차 공격을 위한 정보를 제공하는 취약점.
- 악성 콘텐츠(CS) : 웹 애플리케이션에 정상 콘텐츠 대신 악성 콘텐츠를 주입하여 사용자에게 악의적인 영향을 미치는 취약점.
- 약한 문자열 강도(BF) : 사용자 이름·패스워드·신용 카드 정보·암호화 키 등을 자동으로 대입하여 여러 시행착오 끝에 맞는 값을 발견하는 취약점.
- 취약한 패스워드 복구(PR) : 취약한 패스워드 복구 메커니즘으로 공격자가 다른 사용자의 패스워드를 불법으로 획득·변경·복구할 수 있는 취약점.
- 크로스사이트 리퀘스트 변조(CSRF) : 로그온한 사용자 브라우저로 하여금 세션 쿠키와 인증 정보를 포함하는 위조된 HTTP 요청을 취약한 웹 애플리케이션에 전송하는 취약점.
- 세션 예측(SE) : 단순히 숫자가 증가하는 등의 취약한 세션 식별자(ID)를 예측하여 세션을 가로챌 수 있는 취약점.
- 불충분한 인가(IN) : 민감한 데이터 또는 기능에 대한 접근 권한 제한을 두지 않은 취약점.
- 불충분한 세션 만료(SC) : 세션 만료 기간을 정하지 않거나 너무 길게 설정하여 공격자가 만료되지 않은 세션을 활용할 수 있게 되는 취약점.
- 세션 고정(SF) : 세션값을 고정하여 명확한 세션 식별자(ID)값으로 사용자가 로그인해 정의된 세션 식별자가 사용 가능하게 되는 취약점.
- 자동화 공격(AU) : 웹 애플리케이션의 정해진 프로세스에 자동화된 공격을 수행하여 자동으로 수많은 프로세스가 진행되는 취약점.
- 프로세스 검증 누락(PV) : 공격자가 응용의 계획된 흐름 통제를 우회하는 것을 허가하는 취약점.
- 파일 업로드(FU) : 파일 업로드 기능을 이용해 시스템 명령어를 실행할 수 있는 웹 프로그램을 업로드할 수 있는 취약점.
- 파일 다운로드(FD) : 파일 다운로드 스크립트를 이용해 첨부된 주요 파일을 다운로드할 수 있는 취약점.
- 관리자 페이지 노출(AE) : 단순한 관리자 페이지 이름이나 설정·프로그램 설계상의 오류로 관리자 메뉴에 직접 접근할 수 있는 취약점.
- 경로 추적(PT) : 외부에서 디렉터리에 접근하는 것이 허가되어 웹 루트 디렉터리에서 외부의 파일까지 접근하고 실행할 수 있는 취약점.
- 위치 공개(PL) : 예측 가능한 디렉터리나 파일명을 사용하여 위치가 쉽게 노출되어 공격자가 민감한 정보가 담긴 데이터에 접근할 수 있게 되는 취약점.
- 데이터 평문 전송(SN) : 서버와 클라이언트 간 통신 시 암호화하여 전송하지 않아 중요 정보 등이 평문으로 전송되는 취약점.

#### 네트워크 취약점 진단 방법
- ping 스캔 : Ping 스캐닝 기술로 점검 대상 IP 대역에서 운용 중인 네트워크 장비·시스템을 식별하고 그 정보로 네트워크 구조를 파악하는 진단 방법.
- TCP/UDP 포트 스캔 : 포트에 연결하여 시스템이 동작하고 있는지와 실행·Listening 중인 서비스를 파악하는 진단 방법.
- 패스워드 추측 : username/passwd 목록 등을 이용해 자동으로 패스워드 추측 공격을 수행하여 패스워드 취약점을 발견하는 진단 방법.
- 패킷 스니퍼(sniffer) : 일반 사용자 계정의 홈 디렉터리에 설치되어 로컬 네트워크상의 트래픽을 감시하는 진단 방법.
- 원격 서비스 점검 : 네트워크에 대한 불법적 접근이나 비인가된 원격 서비스 접속을 점검하는 진단 방법.
- 백도어 점검 : 시스템 설계자나 관리자에 의해 고의로 남겨진, 응용 프로그램이나 운영 체제에 삽입된 프로그램 코드를 점검하는 진단 방법.
- OID(Object Identifiers) : SNMP 에이전트를 통해 얻을 수 있는 시스템 정보로, 포트의 개폐 여부나 IP·MAC 같은 사용자 정보를 획득할 수 있음.

### 2-2. 정보 시스템 취약점 진단 수행

#### 서버 취약점 진단 도구
- 개념 : 시스템 기반의 보안 취약점을 점검하는 도구.
- 특징 : 대상 호스트에 에이전트 설치로 단일 시스템을 고밀도 점검, 운영 체제별 점검 내용 영향(취약점 점검 DB 구축), 로컬 접근(root 계정) 필요.
- 동작 원리 : 점검 규칙 설정 → 규칙에 따라 테스트 → 결과 보고.
- 구조 : 로컬 설치(소수 시스템), 매니저-에이전트 2-Tier(원격 점검·암호화 통신·중앙 통제), 콘솔-매니저-에이전트 3-Tier(다수 관리자·대규모 환경).

#### 네트워크 취약점 진단 도구
- 개념 : 시스템·네트워크상의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 복구 보고서 제공, 비인가 장비·접속점 탐지, OS·실행 서비스 정보 제공, 에이전트 설치 불필요, 공격자 기술 모방.
- 동작 원리 : 시스템 탐지(Ping) → 서비스 포트 탐지(포트 스캔) → 점검 모듈로 취약점 점검 → 보호 대책 제시.

#### 애플리케이션 취약점 진단 도구
- 개념 : 애플리케이션의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 애플리케이션 코드 점검, 가능성 있는 취약점·에러 발견, 모의 해킹 프로세스 접목, 보호 대책 제시.

#### 데이터베이스 취약점 진단 도구
- 개념 : DB의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 에이전트·컴포넌트 설치 불필요(IP·포트 정보만으로 점검), 국제 보안 기준 기반 다각적 분석, 사용자 정의 진단 지원.

#### 애플리케이션 취약점 진단 점검
- 중요 정보 평문 저장 : 패스워드·개인 정보·사용자 권한 정보 등 중요 정보를 암호화하지 않고 평문으로 저장하여 발생하는 약점으로, 저장·사용 시 반드시 암·복호화 과정을 거쳐야 함.
- 초기화되지 않은 변수 사용 : 변수의 선언과 동시에 초기화가 이루어지지 않아 발생하는 약점으로, 객체 필드가 초기값을 갖지 않는 경우 생성자를 통한 초기화 여부를 확인.

#### 위험도에 따른 대책 선정
- 개념 : 수용 가능한 목표 위험 수준보다 위험도가 높을 경우 선택해야 하는 대책으로, 위험도가 높은 개별 자산 또는 자산 그룹에 대하여 별도로 적용.
- 주요 대책 영역 : 정책 문서의 유형, 외부 전문가 활용, 계약·서비스 수준 협약 보안 관리, 외부자 보안 실행 관리, 정보 자산의 분류 및 취급, 주요 직무 담당자 관리, 비밀 유지, 데이터센터 보안(위치·구조·출입 통제), 장비 보호(주요 시스템 보호·배치·안전한 폐기), 분석 및 설계 보안 관리, 구현 및 이행 보안 관리, 변경 관리, 암호 정책·사용·키 관리, 특수 권한 관리, 운영 절차와 책임, 네트워크 운영, 매체 및 문서 관리, 법적 요구 사항 준수 검토, 모니터링.
