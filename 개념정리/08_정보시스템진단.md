# 정보 시스템 진단 — 개념 정리

> 학습모듈: 정보 시스템 진단 기준을 마련하고, 진단 결과를 도출하며, 대응책을 마련한다.

## 학습 1. 정보 시스템 진단 기준 마련하기

### 1-1. 정보 시스템 진단 준비 및 취약점 정보 파악

#### 정보 시스템의 개요
- 개념 : 정보 시스템은 기업이 다양한 의사 결정을 내리는 데 필요한 정보를 수집하고 그것을 목적에 따라 가공하며, 그 결과를 의사 결정자에게 제공하는 구조이다.
- 정보 시스템의 관리 요소(10가지) :
  - 구성 및 변경 관리 : HW·SW의 구성 현황·이력·구성 파일·파라미터·구성도를 관리하고 변경 사항을 효율적으로 관리하는 업무.
  - 운영 상태 관리 : 구성 요소의 운영 상태를 관리해 이상 징후를 발견·기록·분류·통지하여 가용성을 향상.
  - 성능 관리 : 효율·응답 속도를 최적 상태로 유지하기 위해 낮은 성능 요소를 찾아 개선.
  - 장애 관리 : SW·HW·DB·네트워크의 고장·장애·서비스 불능을 관찰·진단·보고·제어·처리하는 과정.
  - 보안 관리 : 정보 자산의 정보 보호 활동·절차를 체계적으로 관리해 기밀성·무결성·가용성을 확보하고 불법 유출·파괴·변경으로부터 보호.
  - 백업 관리 : 장애·재해로 정보 소실·손상에 대비해 데이터를 별도 매체에 예비 저장하여 정보 시스템의 영속성을 보장.
  - 사용자 지원 관리 : 운영 지침서 수정·보완, 교육, 요구 처리 등으로 최종 사용자 만족도를 향상.
  - 전산실 관리 : 전산실 접근 통제와 건물·설비 관리로 전산 시스템 운용의 안전성·신뢰성을 보장.
  - 운영 아웃소싱 관리 : 효율적·체계적 운영을 위해 외부 기관으로부터 운영 서비스를 획득하는 절차.
  - 예산 관리 : 운영 예산 소요를 예측하고 예산을 계획·집행·통제하는 절차와 지침.

#### 위험의 정의
- 개념 : 위험(risk)은 원하지 않는 사건이 발생하여 손실 또는 부정적 영향을 미칠 가능성. 손실 사건 발생 가능성과 발생 손실 정도에 비례.
- 위험 평가 산식 : 위험 = 발생 가능성 × 손실의 정도.
- 위험 평가 함수 : 위험 = f(자산, 위협, 취약성).

#### 위험의 구성 요소
- 자산(Asset) : 조직이 보호해야 할 대상(정보·HW·SW, 인력 등 무형 자산 포함). 데이터·문서·SW·HW·물리적·인적 자산 등으로 구분. 위험 분석 시 보안 사고 발생 시의 손실(loss)을 파악.
- 위협(Threats) : 자산에 손실을 초래할 수 있는 원치 않는 사건의 잠재적 원인. 환경적 요인(자연재해·장비 고장)과 인간에 의한 위협(의도적/우연적)으로 분류. 핵심 속성은 발생 가능성(연간 발생 횟수). 위협의 발생 가능성과 성공 가능성은 다를 수 있음.
- 취약성(Vulnerability) : 자산의 잠재적 속성으로 위협의 이용 대상, 또는 정보 보호 대책의 미비. 자산과 위협 사이의 관계를 맺어 주는 특성.
  - 취약성 속성 : 모든 자산에 잠재, 그 자체로는 위협을 초래하지 않고 위협에 이용될 때만 위험 발생, 대응책이 늘수록 감소, 대응책 자체도 취약성을 가져 0이 될 수 없음.
  - 취약성 유형 분류 : 관리적 측면(정책·조직·인사), 물리적 측면(건물·시설물), 기술적 측면(네트워크·시스템·DB·PC·응용 프로그램).

### 1-2. 정보 시스템 위험 평가 방법 및 보호 대책 파악

#### 위험 평가 방법론
- 위험 관리 : 자산의 위험을 수용 가능한 수준으로 유지하기 위해 위험을 분석하고 보호 대책을 마련하는 과정.
- 기준선 접근법(Baseline) : 모든 정보 자산에 보호 기본 수준을 정하고 보호 대책을 적용. 시간·비용 적게 들지만 부서별 보안 수준이 과대/과소할 수 있음.
- 비정형 접근법(Informal) : 전문가의 지식·경험으로 위험 분석(전문가 판단법). 작은 조직에 효과적이나 위험 평가·비용 도출이 어렵고 사후 관리 제한.
- 상세 위험 분석(Detail Risk Analysis) : 자산 가치·위협·취약성을 분석해 위험 정도를 결정. 적절한 보안 수준 마련 가능하나 전문 지식·시간·노력 소요.
  - 정성적 위험 평가 : 손실·위험을 개략적 크기로 비교, 상황 설명으로 묘사.
  - 정량적 위험 평가 : 위험을 손실액 등 숫자값으로 표현. 연간 기대 손실 = 위협 성공 시 예상 손실액 × 발생률.
- 복합 접근법(Combined) : 위험이 높은 시스템은 상세 위험 분석, 그 외는 기준선 접근법 적용. 신속한 보안 전략 수립·효율적.

#### 위험 처리 방법
- 위험 수용 : 현재의 위험을 받아들이고 잠재적 손실 비용을 감수(일정 수준 이하 위험 인정).
- 위험 감소 : 위험을 감소시키는 대책을 채택·구현. 비용 효과 분석으로 양(+)의 효과를 갖는 대책 선택(보호 대책 효과 = 원래 ALE − 대책 구현 후 ALE − 연간 대책 비용).
- 위험 회피 : 위험이 존재하는 사업을 수행하지 않고 포기.
- 위험 전가 : 보험·외주 등으로 잠재적 위험 비용을 제3자에게 이전.

#### 정보 보호 관리 체계
- 개념 : 조직 자산을 효율적으로 보호하도록 기술적·물리적·관리적·조직적 보호 대책을 구현하고 지속 관리·운영하는 종합적 체계. 기밀성·무결성·가용성·준거성 달성을 위해 위험을 평가하고 대책을 수립·운영.
- 관리 과정(5단계) : 정보 보호 정책 수립 → 정보 보호 관리 체계 범위 설정 → 위험 관리 → 구현 → 사후 관리.

## 학습 2. 정보 시스템 진단 결과 도출하기

### 2-1. 정보 시스템 취약점 진단 방법 파악

#### 서버 취약점
- 계정 관리 : root 계정 원격 접속 제한, root 계정 전환(su) 제한(무작위 대입·추측 공격 방지), root 외 동일 UID 금지.
- 파일 및 디렉터리 관리 : root 홈·PATH 권한 설정(PATH에 "." 포함 시 비의도적 명령 실행), 파일·디렉터리 소유자 설정, UMASK 설정 관리, 숨겨진 파일·디렉터리 검색·제거.
- 서비스 관리 : r 계열 서비스 비활성화, automountd 제거(RPC 악용 방지), DoS 취약 서비스 비활성화, RPC 서비스 확인(버퍼 오버플로 취약).

#### 네트워크 취약점
- 계정 관리 : 패스워드 복잡성 설정(무작위 대입 방지), 초기 설정 패스워드 변경.
- 접근 관리 : 불필요한 보조 입출력 포트 사용 금지, vty 접속 시 안전한 프로토콜 사용(Telnet은 평문 전송으로 스니핑에 노출).
- 패치 관리 : 최신 보안 패치 및 벤더 권고 적용.
- 로그 관리 : 정책에 따른 로깅 설정, 로깅 버퍼 크기 설정(초과 시 로그 손실).
- 기능 관리 : SNMP 서비스 확인(UDP 기반 공격 취약), Spoofing 방지 필터링 적용.

#### 애플리케이션 취약점
- 부적절한 에러 처리 : 다양한 에러를 유발해 에러 결괏값으로 웹 프로그램 구조·환경 설정을 추측해 공격.
- SQL 인젝션 : 입력값 유효성 검증 누락 시 조작된 SQL 쿼리로 인증 우회·정보 유출·시스템 명령 실행.
- 불충분한 인증 : 민감 데이터 경로의 인증 절차 부족으로 권한 외 페이지 접근·정보 유출·변조.
- 크로스사이트 스크립팅(XSS) : 게시판 등에 스크립트를 삽입해 열람자 브라우저에서 실행, 인증 정보 유출·악성코드 설치.
- 쿠키 변조 : 보호되지 않은 쿠키값 변조로 타 사용자 위장·권한 상승.
- 파라미터 변조 공격 : 클라이언트-서버 간 파라미터값을 조작해 중요 정보 조작·권한 획득.

#### 데이터베이스 취약점
- 계정 관리 : 기본 계정(system·scott 등) 패스워드·정책 변경, 불필요 계정(SCOTT·ADAMS 등) 제거·잠금, DBA 권한(SYSDBA)을 필요 계정에만 허용.
- 접근 관리 : 원격 접속 제한(미허용 IP 차단), DBA 외 시스템 테이블 접근 차단, 주요 설정·패스워드 파일 접근 권한 적절 설정.
- 옵션 관리 : 응용/DBA 계정 Role의 Public 설정 방지, OS_ROLES·REMOTE_OS_AUTHENT·REMOTE_OS_ROLES를 FALSE로 설정.
- 패치 관리 : 최신 보안 패치·벤더 권고 적용, 접근·변경·삭제 감사 기록을 정책에 적합하게 설정.

### 2-2. 정보 시스템 취약점 진단 수행

#### 서버 취약점 진단 도구
- 개념 : 시스템 기반의 보안 취약점을 점검하는 도구.
- 특징 : 대상 호스트에 에이전트 설치로 단일 시스템을 고밀도 점검, 운영 체제별 점검 내용 영향(취약점 점검 DB 구축), 로컬 접근(root 계정) 필요.
- 동작 원리 : 점검 규칙 설정 → 규칙에 따라 테스트 → 결과 보고.
- 구조 : 로컬 설치(소수 시스템), 매니저-에이전트 2-Tier(원격 점검·암호화 통신·중앙 통제), 콘솔-매니저-에이전트 3-Tier(다수 관리자·대규모 환경).

#### 네트워크 취약점 진단 도구
- 개념 : 시스템·네트워크상의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 복구 보고서 제공, 비인가 장비·접속점 탐지, OS·실행 서비스 정보 제공, 에이전트 설치 불필요, 공격자 기술 모방.
- 동작 원리 : 시스템 탐지(Ping) → 서비스 포트 탐지(포트 스캔) → 점검 모듈로 취약점 점검 → 보호 대책 제시.

#### 애플리케이션 취약점 진단 도구
- 개념 : 애플리케이션의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 애플리케이션 코드 점검, 가능성 있는 취약점·에러 발견, 모의 해킹 프로세스 접목, 보호 대책 제시.

#### 데이터베이스 취약점 진단 도구
- 개념 : DB의 취약점을 발견하고 자동 진단하는 도구.
- 특징 : 에이전트·컴포넌트 설치 불필요(IP·포트 정보만으로 점검), 국제 보안 기준 기반 다각적 분석, 사용자 정의 진단 지원.

## 학습 3. 정보 시스템 대응책 마련하기

### 3-1. 정보 시스템 취약점 위험 대응 계획 수립

#### 정보 보호 계획 수립을 위한 프로젝트 구성
- 개념 : 위험을 수용 수준으로 감소시킬 대책 마련 후, 유사 대책을 프로젝트로 통합하고 우선순위·일정을 수립.
- 즉시 교정 가능한 취약점 제거 : 구성 설정 변경·권한 변경·취약 패스워드 변경 등 빠른 처리(책임자·완료 시한·확인 절차 필요, 변경 영향도 파악).
- 정책 및 절차 수립 : 정책을 먼저 수립하고 준수 절차를 마련, 이해관계자의 이해·동의 확보.
- 정보 보호 시스템의 도입 및 관련 교육 : 정보 보호 정책 요구 사항 반영, 통제 제거 부작용 고려, 이해관계자 교육 수행.
- 모니터링 및 감사 : 실제 수행은 사후 관리 단계지만 정책·절차·책임·시스템 도입은 계획 수립 단계에서 결정. 감사 사항은 체크리스트·감사 요구 사항으로 제공.

#### 정보 보호 구현 계획의 수립
- 정보 보호 계획의 수립 : 프로젝트별 예산·일정·세부 내용·책임을 포함. 영향받는 부서 책임자·실무자가 참여(정보보호위원회 검토 권장).
- 정보 보호 계획의 승인 : 최고 경영진·정보 보호 책임자 등 최종 의사 결정권자가 명시적으로 승인.

#### 정보 보호 대책 명세서 작성
- 개념 : 모든 통제 사항에 대해 선택 내용과 선택하지 않은 이유를 명시한 문서. 전반적 대책 현황 파악·타 조직 협정 등에 사용.
- 포함 사항 : 선정된 대책의 명세, 구현 확인 근거, 선정되지 않은 대책 목록, 선정되지 않은 근거.

### 3-2. 정보 시스템 보호 대책의 적용

#### 정보 보호를 위한 필수 대책 선정
- 개념 : 정보 자산을 성공적으로 보호·관리하기 위해 기본적으로 구현해야 하는 일반적 대책.
- 주요 대책 영역 : 정책(승인·공표·체계·유지 관리), 조직(체계·책임과 역할), 정보 자산 조사 및 책임 할당, 교육 및 훈련, 인사 보안(책임 할당·적격 심사·비밀 유지 서약), 물리적 보안(보호 구역·접근 통제·데이터센터), 정책 준수 검토·사후 관리, 업무 연속성 관리(체계 수립·계획 수립·구현·시험·유지 관리).

#### 자산의 종류에 따른 대책 선정
- 개념 : IT 자산 중 하나라도 존재하면 필수적으로 구현해야 하는 일반적 대책.
- 주요 대책 : 장비 보호(전원 공급·케이블 보호·장비 보수), 접근 통제(접근 통제 정책·방법·문서화), 사용자 접근 관리(등록·패스워드·권한 검토), 운영 접근 통제(운영 체제 접근·응용 프로그램 접근·DB 접근·네트워크 접근).
